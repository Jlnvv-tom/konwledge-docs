# 第14章 Redis高可用架构：哨兵机制

在分布式系统中，单节点的Redis服务始终面临单点故障风险。当主节点因硬件故障、进程崩溃或网络抖动而不可用时，如果完全依赖人工介入来恢复服务，系统的可用性将大打折扣。Redis从2.8版本引入的哨兵机制，正是为了解决这一痛点而生的。哨兵是Redis官方推荐的高可用解决方案，它以独立的进程运行，通过主动监控、自动故障转移和客户端配置更新三大能力，构建起一套完整的Redis高可用保障体系。本章将系统讲解哨兵的核心原理、配置参数、集群搭建以及故障转移的全流程实操，帮助你从零到一掌握Redis哨兵的高可用架构设计。

## 14.1 哨兵的核心作用：主节点故障自动切换

在传统的Redis主从架构中，主节点承担所有写操作。一旦主节点发生故障，整个系统将丧失写能力，即使存在多个从节点可以提供读服务也无济于事。人工介入需要经历发现问题、确认故障、选择新主节点、修改从节点复制目标、通知客户端等一系列操作，在生产环境中可能耗时数分钟乃至更久。对于对可用性要求较高的在线业务而言，这段时间的服务中断往往是不可接受的。

哨兵机制的本质是一组运行在独立进程中的分布式协调服务。它不存储业务数据，专注于监控Redis集群的运行状态。当它检测到主节点不可达时，会自动发起故障转移流程：从从节点中选举出新的主节点，将其他从节点重新指向新主节点，并更新客户端连接配置。整个过程对应用透明，业务中断时间可以控制在秒级。

需要特别说明的是，哨兵本身也面临单点故障问题。因此生产环境中必须部署多个哨兵组成集群，利用哨兵之间的相互监控来确保整个高可用系统本身的可靠性。哨兵解决的核心问题有三个维度：第一是故障检测，哨兵持续向主从节点发送PING命令，如果节点在指定时间内未响应则将其标记为主观下线。第二是自动故障转移，当主节点被判定为主观下线后，哨兵集群通过类似Raft的协议协商出一致意见，将某个从节点晋升为新的主节点。第三是配置更新，故障转移完成后，哨兵通过发布订阅机制通知客户端新的主节点地址，客户端无需人工干预即可自动重连。

这里有一个常见的误解需要澄清：哨兵和Redis Cluster解决的是不同层面的问题。哨兵解决的是主从复制架构下的自动故障切换，每个分片仍然是主从结构，哨兵只负责监控和切换，不负责数据分片。Redis Cluster则是3.0版本引入的数据分片方案，通过哈希槽将数据分布到多个主节点上。如果你需要存储TB级别的数据，单个主节点的内存无法承载，那么Redis Cluster是更合适的路径；如果你对可用性要求很高且数据量不大（通常几十GB以内），哨兵是一个简洁有效的选择。

## 14.2 哨兵机制原理：监控、通知与故障转移流程

### 14.2.1 故障检测：主观下线与客观下线

哨兵对节点健康状况的判断分为两个阶段。第一阶段是主观下线（Subjective Down，简称SDOWN）：哨兵按照配置的down-after-milliseconds参数向被监控节点发送PING命令。如果节点在指定毫秒数内没有返回有效的PONG响应，哨兵就将该节点标记为主观下线。这个判断是单个哨兵独立做出的，不涉及其他哨兵的意见。

第二阶段是客观下线（Objectively Down，简称ODOWN）：当主节点被标记为SDOWN后，哨兵需要确认其他哨兵也认为主节点不可达。这一步通过SENTINEL is-master-down-by-addr命令在哨兵之间交换信息。如果达到quorum参数指定数量的哨兵都认为主节点已下线，那么主节点就被标记为客观下线。只有进入ODOWN状态后，哨兵才会触发故障转移流程。

从节点的下线检测规则稍有不同。从节点只做主观下线判断，不进入客观下线流程，因为从节点故障不需要触发整个集群的故障转移。这是为了避免因为单从节点故障而触发全集群的故障转移操作。Sentinel节点之间通过发布订阅机制相互感知，当一个Sentinel发现另一个Sentinel不可达时，会将其标记为Sentinel主观下线，但不参与客观下线的投票。

一个常见的踩坑点是down-after-milliseconds参数设置过短导致频繁误判。在网络不稳定的测试环境或使用虚拟机部署的生产环境中，如果将该值设置为1000毫秒，网络抖动很容易触发主观下线，进而触发不必要的故障转移。建议在生产环境中根据网络状况逐步调优，一般从5000毫秒起步，观察是否出现误判后再做调整。

### 14.2.2 领导者选举：Raft风格的投票过程

当主节点被判定为客观下线之后，故障转移工作不能由多个哨兵同时进行，否则会产生脑裂问题——多个从节点被同时提升为新主节点，数据不一致。Redis哨兵采用类似Raft算法的领导者选举机制，确保只有一个哨兵来主导整个故障转移过程。

选举的触发条件是：主节点被标记为客观下线，且当前哨兵没有被其他哨兵选为领导者。选举过程由希望执行故障转移的哨兵发起，它首先向所有其他哨兵发送SENTINEL is-master-down-by-addr命令，携带自己的runid和配置epoch。如果一个哨兵还没有选择领导者，它会回复同意并将epoch更新为请求者携带的epoch。一旦某个哨兵获得了多数哨兵的同意票，它就成为本次故障转移的领导者，获得执行故障转移的权限。

配置epoch在哨兵选举中扮演关键角色。每次故障转移都会生成一个新的配置epoch，格式为时间戳乘以1000加上毫秒内的计数器。这个值保证了不同哨兵提出的配置在时间上的全序关系。当多个哨兵竞争时，epoch更大的方案会覆盖epoch更小的方案。即使某个哨兵收到了多数同意票，在它实际执行故障转移之前，其他哨兵仍可能因为网络分区而继续发送选举请求。因此哨兵在执行故障转移前会再次检查自己是否仍然是当前epoch下的领导者。

### 14.2.3 故障转移流程详解

被选为领导者的哨兵开始执行故障转移，整个过程分为三个步骤：选择新主节点、从节点指向新主节点、撤销旧主节点的从属关系。

第一步是选择新主节点。哨兵从当前主节点的所有从节点中挑选，规则按以下优先级排序：首先排除不稳定的从节点——即与主节点断开连接超过down-after-milliseconds乘以10毫秒的从节点，因为这类从节点数据可能已严重滞后。然后在剩余的稳定从节点中，优先选择replica-priority配置最大的节点（该值默认为100，可通过配置文件修改）。如果优先级相同，则比较从节点的复制偏移量，选择数据最新的从节点。如果复制偏移量也相同，则选择runid字典序最小的从节点。

第二步是向选中的从节点发送SLAVEOF NO ONE命令，将其提升为独立主节点。哨兵会持续监控这个新主节点的状态，确保它正常对外提供服务。

第三步是更新其他从节点的复制目标。哨兵向所有其他从节点发送SLAVEOF命令，让它们开始复制新的主节点。这个过程并行执行，由parallel-syncs参数控制并发度。

最后一步是撤销旧主节点的从属关系。当旧主节点恢复后，哨兵会向其发送SLAVEOF命令，使其变为新主节点的从节点，而不是简单地丢弃它。这是一个容易被忽略的设计细节——故障转移后旧主节点应该被视为数据源之一重新纳入体系，确保数据不会因为一次故障而彻底丢失。

### 14.2.4 配置传播与客户端通知

故障转移完成后，领导者哨兵需要将新的配置信息同步给所有其他哨兵和客户端。哨兵集群内部通过发布订阅机制来传播配置：每次配置变更都会生成一个新的配置epoch，哨兵通过PUBLISH命令将新的主节点信息广播到_sentinel_:hello频道。其他哨兵收到消息后更新自己的本地配置，并确认epoch是否更新。

对于客户端而言，哨兵提供了SENTINEL get-master-addr-by-name命令来查询当前主节点的地址。客户端在连接Redis时应该使用哨兵提供的地址，而不是直接硬编码主节点IP和端口。主流Redis客户端库（Jedis、lettuce、StackExchange.Redis等）都原生支持哨兵模式，它们内部会订阅哨兵的配置变更频道，当收到主节点变更通知时自动重置连接池。

这里有一个实战中经常遇到的问题：客户端连接池未正确处理哨兵切换。很多开发者在初次集成哨兵时，虽然配置了哨兵地址列表，但连接池初始化时使用的是启动时查询到的第一个主节点地址。当哨兵完成故障转移后，连接池仍然持有指向旧主节点的连接，导致写操作失败。正确的做法是让客户端始终通过哨兵来获取主节点地址，主流Redis客户端库（Jedis、lettuce、StackExchange.Redis等）只需配置sentinelMasterId和sentinelAddresses两个参数，即可自动处理地址查询和切换逻辑。

## 14.3 哨兵配置：sentinel.conf关键参数详解

### 14.3.1 基础配置参数

sentinel.conf是哨兵进程的核心配置文件。理解每个参数的作用，是正确部署和调优哨兵集群的前提。

首先是基础连接参数：

```conf
port 26379
daemonize no
logfile ""
pidfile ""
```

port指定哨兵进程监听的端口，默认为26379，如果在一台机器上部署多个哨兵，需要为每个哨兵分配不同端口。daemonize控制哨兵是否以后台进程运行，建议设置为yes以配合supervisor或systemd管理。logfile指定日志输出路径，生产环境应该指向一个具体的文件路径以便排查问题。

然后是监控主节点的核心参数：

```conf
sentinel monitor mymaster 127.0.0.1 6379 2
sentinel down-after-milliseconds mymaster 30000
sentinel parallel-syncs mymaster 1
sentinel failover-timeout mymaster 180000
```

sentinel monitor的参数格式为：sentinel monitor <master-name> <ip> <port> <quorum>。其中quorum参数指定了多少个哨兵认为主节点下线才触发客观下线判断。例如部署了3个哨兵，quorum设置为2，那么需要至少2个哨兵认为主节点下线才会触发故障转移。这个值通常设置为哨兵总数的多数即可。如果设置为1，任何一个哨兵都能触发故障转移，存在误触发风险；如果设置为哨兵总数，则所有哨兵都必须认为下线才触发，风险是任何一个哨兵出问题就会阻止故障转移。

down-after-milliseconds指定了主观下线的判定时间。当PING命令在指定毫秒内没有收到有效响应时，就将该节点标记为主观下线。需要同时为主节点和从节点设置这个值。从节点的主观下线判断主要用于决定是否将其排除在新主节点的候选列表之外。

### 14.3.2 故障转移参数

parallel-syncs控制了故障转移时从节点并行同步的并发度。当新主节点被选举出来后，哨兵会同时向多个从节点发送SLAVEOF命令让它们复制新主节点。如果从节点数量较多，parallel-syncs的值过大会对新主节点造成较大的复制压力，可能导致新主节点因内存不足而崩溃。生产环境建议将这个值设置为1或2，确保每次只有一个从节点在同步数据。同步过程中从节点不可用，但比起因为复制风暴导致新主节点崩溃，这个代价是值得的。

failover-timeout指定了故障转移的超时时间，默认为180000毫秒（3分钟）。这个参数影响的是故障转移各个环节的超时判断：选择新主节点超时、从节点指向新主节点超时、撤销旧主节点从属关系超时等。每次超时后都会重新尝试故障转移，直到达到最大重试次数。因此这个值应该设置为足够容纳一次完整故障转移流程的时间。

replica-priority参数写在Redis节点的redis.conf中而非哨兵配置文件，但它对哨兵的故障转移决策至关重要。这个值默认为100，范围是0到1000。设置为0的节点将永远不会被选为新主节点。当多个从节点优先级相同时，哨兵会比较复制偏移量。这个参数给了运维人员灵活控制故障转移结果的能力——比如可以给一台网络条件更好、硬件配置更高的从节点设置更高的优先级，确保它总是优先晋升。

### 14.3.3 认证与安全参数

生产环境中Redis通常需要设置密码认证，哨兵也必须相应配置才能正常访问Redis节点：

```conf
sentinel auth-pass mymaster MyStrongPassword
sentinel auth-user mymaster redisuser
```

sentinel auth-pass设置了访问主节点所需的密码。如果主从节点的密码相同（这是推荐的实践），只需要配置这一个参数即可。

还有两个与脚本通知相关的参数：

```conf
sentinel notification-script mymaster /var/redis/sentinel_notify.sh
sentinel client-reconfig-script mymaster /var/redis/reconfig.sh
```

notification-script指定了一个脚本路径，当哨兵检测到重要事件（客观下线、故障转移开始/结束等）时会执行该脚本。这个脚本常用于发送告警通知，比如调用钉钉Webhook或发送邮件。这个脚本的执行时间不能超过failover-timeout，否则会阻塞故障转移流程。如果不需要通知功能，可以不配置此项。

client-reconfig-script指定了当故障转移完成后需要执行的脚本。这个脚本接收的参数包括新主节点的地址和端口，以及从节点列表变化情况。它可以用于通知应用层刷新配置或更新负载均衡器的后端列表。

### 14.3.4 常见配置踩坑点

在实际部署中，以下配置错误最为常见。

第一个踩坑点是quorum值设置不当导致故障转移不触发或过度触发。如果部署了5个哨兵但quorum设置为3，当其中2个哨兵因网络问题与集群断开时，剩下的3个哨兵仍可触发故障转移——这没有问题。但如果哨兵总数为3且quorum也设置为3，任何一个哨兵出问题都会阻止故障转移，这是应该避免的配置。正确的做法是quorum设置为哨兵总数除以2加1向下取整。

第二个踩坑点是多个master使用了相同的master-name。在同一台服务器上运行多个Redis主从集群时，每个集群应该使用不同的名称。如果两个集群的master-name相同，哨兵会尝试用同一套参数管理两个集群的节点，导致混乱。

第三个踩坑点是notification-script没有设置可执行权限或路径不存在。哨兵在启动时会检查notification-script文件是否存在，如果文件不存在或权限不足，哨兵可能会拒绝启动或跳过通知逻辑。使用自定义脚本时一定要确认文件存在、权限正确（755或744），并且脚本的shebang行指向有效的解释器。

## 14.4 哨兵集群搭建：多哨兵部署与高可用保障

### 14.4.1 部署架构规划

哨兵集群的部署架构直接影响整个Redis高可用系统的心脏跳动质量。在规划部署时，需要考虑三个核心原则：哨兵进程数量应为奇数、哨兵节点应与数据节点错开部署、哨兵之间网络延迟应在可接受范围内。

哨兵数量为奇数是为了避免平票无法决策的情况。最常见的部署方案是3个哨兵，对应3个Redis节点（一主两从）或6个节点（两主四从）。3个哨兵的配置既能保证多数票决策的可靠性，部署成本也相对可控。5个哨兵可以容忍同时2个节点故障，但部署成本翻倍，一般只在对可用性要求极高的核心系统中使用。

哨兵进程应该部署在与Redis数据节点不同的服务器上。很多人为了节省资源在同一个服务器上同时部署Redis和Sentinel，虽然管理起来简单，但这样做等于把鸡蛋放在同一个篮子里——服务器宕机时不仅Redis主节点不可用，监控它的哨兵也会同时下线，整个高可用体系瞬间失效。理想的方案是将哨兵分散到3台不同的服务器上，每台运行一个哨兵进程，监控另一台服务器上的Redis节点。

网络层面的考量同样重要。哨兵之间用于交换信息的通信是频繁且低延迟的，理想情况下哨兵之间的RTT应在10毫秒以内。如果哨兵分布在不同的数据中心或可用区，跨地域的网络延迟可能导致主观下线误判和领导者选举延迟增加。在这种跨地域场景下，可以适当增大down-after-milliseconds参数来减少误判，但代价是故障检测和切换时间的延长。

### 14.4.2 多哨兵配置文件模板

每个哨兵的配置文件基本相同，只需要调整监听的端口和日志文件路径即可。以下是一份生产环境可用的sentinel.conf模板：

```conf
port 26379
daemonize yes
dir /var/redis/sentinel
logfile /var/log/sentinel.log

sentinel monitor mymaster 10.0.1.10 6379 2
sentinel down-after-milliseconds mymaster 30000
sentinel parallel-syncs mymaster 1
sentinel failover-timeout mymaster 180000

sentinel auth-pass mymaster Redis@2024#Pass

sentinel notification-script mymaster /opt/scripts/sentinel_notify.sh
sentinel client-reconfig-script mymaster /opt/scripts/reconfig.sh

sentinel known-replica mymaster 10.0.1.11 6379
sentinel known-replica mymaster 10.0.1.12 6379
sentinel known-sentinel mymaster 10.0.1.20 26379 a1b2c3d4
sentinel known-sentinel mymaster 10.0.1.21 26379 e5f6g7h8
```

known-replica和known-sentinel这两个参数是可选的，但推荐显式配置。它们记录了已知的从节点和哨兵节点信息，在哨兵重启后可以更快地恢复拓扑认知。如果不配置，哨兵会在启动后通过向主节点发送INFO命令来自动发现从节点信息，但显式添加有利于新哨兵快速获取集群拓扑。

日志目录和运行时目录需要预先创建并赋予适当的权限：

```bash
mkdir -p /var/redis/sentinel /var/log/sentinel /opt/scripts
chown -R redis:redis /var/redis/sentinel /var/log/sentinel
chmod +x /opt/scripts/*.sh
```

### 14.4.3 启动与验证

三个哨兵节点应该使用相同的配置文件启动，启动顺序不敏感。启动命令统一使用redis-sentinel命令：

```bash
redis-sentinel /etc/sentinel.conf
# 或者等效的写法
redis-server /etc/sentinel.conf --sentinel
```

启动后，通过以下命令验证哨兵集群状态：

```bash
redis-cli -p 26379 SENTINEL masters
redis-cli -p 26379 SENTINEL master mymaster
redis-cli -p 26379 SENTINEL slaves mymaster
redis-cli -p 26379 SENTINEL sentinels mymaster
```

SENTINEL masters会列出当前哨兵监控的所有主节点。SENTINEL master mymaster会显示主节点的详细信息，包括状态、地址、从节点数量等。SENTINEL slaves mymaster列出所有从节点及其状态。SENTINEL sentinels mymaster则显示集群中其他哨兵节点的信息。

在输出中需要特别关注flags字段，master的flags为m，如果出现s_down或o_down标记说明主节点正在经历故障转移流程；num-slaves字段显示当前有多少个从节点在线；last-ping-reply字段显示最后一次PONG响应距今的毫秒数，如果接近down-after-milliseconds的值说明网络状况可能不太好。

### 14.4.4 节点扩缩容实战

添加新哨兵的步骤非常简洁：准备好配置文件，在新服务器上启动新哨兵进程，配置文件中填写与其他哨兵相同的主节点监控信息即可。哨兵之间通过订阅频道自动发现新节点，无需修改其他哨兵配置。

移除哨兵节点需要更谨慎的操作。首先确认剩余哨兵数量仍能满足quorum要求，否则移除操作会导致故障转移被阻止。然后停止要移除的哨兵进程，确认剩余哨兵的状态正常。如果被移除的哨兵无法正常停止（例如服务器已经宕机），需要等待超时后其他哨兵将其标记为sdown状态。

## 14.5 故障转移实战：主节点下线与从节点晋升流程

### 14.5.1 模拟故障转移：手工触发与观察

故障转移是哨兵机制的核心功能，理解其完整流程对于故障排查至关重要。我们可以通过手工模拟故障来观察哨兵的行为，但在生产环境执行此操作务必谨慎。

首先准备一个测试环境：一主两从三哨兵。启动所有节点后，正常情况下主节点的角色为master，两个从节点的角色为slave。

手工触发故障转移最直接的方式是通过哨兵发送指令：

```bash
redis-cli -p 26379 SENTINEL failover mymaster
```

这条命令会通知当前连接的哨兵立即对mymaster执行故障转移，跳过客观下线的判断流程。执行后，当前主节点会被标记为不可用，哨兵会按照前述的选择规则挑选一个从节点晋升。

故障转移执行后，依次观察三个关键变化：原主节点的角色变为slave；某个从节点晋升为master；其余从节点重新指向新的主节点。可以通过INFO replication命令在各个Redis节点上验证角色变化：

```bash
redis-cli -h <node-ip> -p 6379 INFO replication
```

### 14.5.2 完整流程时序分析

从主节点实际宕机到故障转移完成，整个过程可以拆解为以下步骤。

第一步，故障检测，耗时由down-after-milliseconds控制，默认为30秒。第二步，客观下线确认，耗时数毫秒到数秒不等。如果quorum为2，这个过程通常在数百毫秒内完成。第三步，领导者选举，耗时数百毫秒，通常在1秒内完成。第四步，新主节点选择与晋升，耗时由从节点数据同步状态决定。第五步，从节点重新指向新主节点，并行执行，受parallel-syncs限制。

整个故障转移过程在网络正常、数据同步及时的理想情况下，总耗时约为down-after-milliseconds加上从节点晋升的时间，一般在30到60秒之间。如果从节点数据滞后严重或者网络条件差，时间可能显著延长。

### 14.5.3 故障转移后的数据安全验证

故障转移完成后，必须验证新主节点的数据完整性。这是在生产环境执行故障转移后绝对不能跳过的一步。

首先检查新主节点的角色和从节点复制状态：

```bash
redis-cli -h <new-master-ip> -p 6379 INFO replication
redis-cli -h <new-master-ip> -p 6379 DBSIZE
```

复制状态应该显示role:master，connected_slaves应该等于预期的从节点数量。然后检查所有从节点的复制状态，每个从节点的master_link_status应该为up，master_repl_offset应该持续增长。

一个容易遗漏的验证点是检查旧主节点恢复后的状态。旧主节点重新上线后，哨兵会自动将其降级为从节点并指向新的主节点。连接旧主节点执行INFO replication，确认role为slave，master_host指向新的主节点IP。如果role仍然是master，说明哨兵没有正确处理旧主节点的重新接入，此时应立即检查哨兵日志。

### 14.5.4 常见故障转移异常处理

故障转移过程中可能遇到多种异常情况，理解这些情况的成因和处理方法是运维能力的体现。

第一种异常是从节点晋升失败。如果被选中的从节点在晋升过程中出错，哨兵会在failover-timeout后重新发起故障转移。此时需要检查新晋升节点的从节点日志，确认SLAVEOF NO ONE命令是否正常执行。权限不足通常是因为redis.conf中未正确配置requirepass或masterauth。

第二种异常是数据丢失。故障转移期间如果主节点不可达，主节点上积压的写入数据将无法同步到从节点，这部分数据会丢失。如果业务对数据一致性要求极高，需要启用min-slaves-to-write和min-slaves-max-lag参数来确保写入前有足够数量的从节点已确认接收数据：

```conf
min-slaves-to-write 2
min-slaves-max-lag 10
```

这组配置的含义是只有当至少2个从节点的复制延迟不超过10秒时，才允许主节点接受写操作。这是一种用可用性换一致性的策略，适用于对数据丢失零容忍的场景。

第三种异常是脑裂问题。如果网络分区导致主节点和从节点被分隔在两个网络分区中，一个分区中的从节点被选为新主节点，而原来的主节点仍在另一个分区中提供服务，就会出现两个主节点同时存在的状况。当网络恢复后，哨兵会强制让旧主节点成为新主节点的从节点，但旧主节点上独有的数据将永久丢失。减少数据丢失风险的措施包括启用上述min-slaves参数，以及将down-after-milliseconds设置得足够长以减少误判。

## 14.6 哨兵监控与告警：状态查看与异常通知配置

### 14.6.1 常用哨兵管理命令

掌握哨兵提供的管理命令是日常运维的基础。下面按功能分类介绍最常用的命令。

状态查询类命令用于获取集群的运行状态：

```bash
SENTINEL masters
SENTINEL master <master-name>
SENTINEL slaves <master-name>
SENTINEL sentinels <master-name>
SENTINEL get-master-addr-by-name <master-name>
```

SENTINEL get-master-addr-by-name是客户端集成的核心命令。它返回当前被确认为主节点的Redis实例地址。客户端库在初始化连接时会调用此命令，后续收到配置更新通知时也会重新调用它。

管理操作类命令用于手工干预集群状态：

```bash
SENTINEL failover <master-name>
SENTINEL reset <pattern>
SENTINEL flushconfig
```

SENTINEL flushconfig命令会将当前哨兵的配置状态强制写回sentinel.conf文件。这在手动修改了运行中的哨兵配置后，想要持久化这些修改时非常有用。SENTINEL set命令允许在运行时动态修改配置参数，例如：

```bash
SENTINEL set mymaster down-after-milliseconds 20000
```

动态修改的参数会立即生效，同时会被写入sentinel.conf以确保哨兵重启后仍然保持这些设置。

### 14.6.2 哨兵日志分析

哨兵日志是排查问题和理解集群行为的最重要信息来源。以下是几种典型日志条目的解读。

主节点主观下线：
```
+sdown master mymaster 127.0.0.1 6379
```
+sdown表示主观下线，flags前没有哨兵runid前缀说明这是当前哨兵自己做出的判断。

主节点客观下线：
```
+odown master mymaster 127.0.0.1 6379 #quorum 2/2
```
#quorum 2/2表示有2个哨兵达到配置的quorum数量认为主节点不可达。

领导者选举成功：
```
+elected-leader master mymaster 26379 runid e5f6g7h8...
```
+elected-leader表示当前哨兵在领导者选举中获胜，获得了执行故障转移的权限。

故障转移各个阶段依次出现的日志包括：+failover-state-select-slave（选择从节点）、+failover-state-send-slaveof-noone（发送SLAVEOF NO ONE命令）、+failover-state-promote（从节点晋升完成）、+failover-state-update-config（更新其他从节点配置）。

旧主节点重新加入：
```
+convert-to-slave master mymaster 127.0.0.1 6379
```
旧主节点恢复后被降级为从节点并指向新主节点。这条日志确认了故障转移流程的收尾步骤正确执行。

在排查问题时，建议重点关注日志中连续出现的+/-odown和sdown标记，如果这些标记频繁出现，说明网络可能存在间歇性故障。如果领导者选举日志（+new-leader）短时间内多次出现，说明选举过程可能遭遇问题，例如网络不稳定或者epoch竞争。

### 14.6.3 告警通知配置

仅依靠定期查看哨兵状态来监控集群是不够的，需要配置自动告警机制让系统在异常发生时主动通知运维人员。哨兵提供了notification-script机制来实现这一点。

notification-script会在以下事件发生时执行：主节点或从节点主观下线（+sdown）、客观下线（+odown）、故障转移开始（+failover-start）、故障转移超时（+failover-abort-*）、从节点晋升（+promoted-slave）、配置变更（+config-update-from）等。

一个实用的告警脚本示例：

```bash
#!/bin/bash
NOTIFY_TYPE="$1"
MASTER_NAME="$2"
MASTER_IP="$3"
MASTER_PORT="$4"

case "$NOTIFY_TYPE" in
    +odown|+new-master|+promoted-slave|+failover-start)
        MESSAGE="[Redis哨兵告警] ${NOTIFY_TYPE} 集群: ${MASTER_NAME} 主节点: ${MASTER_IP}:${MASTER_PORT}"
        curl -s "https://oapi.dingtalk.com/robot/send?access_token=YOUR_TOKEN" \
          -H 'Content-Type: application/json' \
          -d "{\"msgtype\":\"text\",\"text\":{\"content\":\"${MESSAGE}\"}}" > /dev/null 2>&1
        ;;
esac
```

这个脚本根据事件类型选择性地发送告警。对于严重事件（客观下线、故障转移开始、从节点晋升）则立即通知。脚本中需要注意的是，必须将告警发送逻辑放在脚本开头执行，如果脚本执行时间过长，会阻塞哨兵的后续操作。建议脚本内部对网络调用设置较短的超时时间，避免影响哨兵主流程。

生产环境中更推荐的方案是结合Prometheus和Grafana进行哨兵集群监控。通过redis_exporter的Sentinel探测功能，可以收集每个哨兵的各项指标：主节点可用性、从节点数量、故障转移次数、领导者选举耗时等。在Grafana中配置告警规则，当可用从节点数量低于阈值或故障转移频率异常升高时自动触发告警。

### 14.6.4 日常巡检与预防性维护

故障往往有征兆可循。日常巡检应该关注以下几个方面，做到早发现早处理。

定期检查主从节点的角色和复制延迟。通过定时任务收集所有节点的INFO replication输出，关注master_link_status字段是否为up，master_repl_offset的增长速度是否正常。如果发现复制延迟持续增大，说明从节点的处理能力跟不上主节点的写入速度，需要考虑扩容从节点数量或优化业务写入模式。

定期检查哨兵节点的运行状态。确认所有哨兵进程都在运行，没有僵尸进程或反复重启的情况。哨兵进程异常退出的原因通常是OOM（内存不足）、配置错误或依赖库缺失。在内存受限的环境中，应该通过systemd或supervisor为哨兵进程设置合理的内存限制（MemoryMax）。

定期审查和测试故障转移流程。每隔一个季度应该进行一次模拟故障转移演练，验证告警通知是否正常、故障转移时间是否符合SLA要求、客户端重连是否顺畅。很多问题只有在实际执行故障转移时才会暴露出来。定期备份sentinel.conf文件，配置文件应该纳入配置管理系统的版本控制中。

## 总结

本章系统讲解了Redis哨兵机制的完整知识体系。我们从哨兵的核心价值出发，理解了它如何通过故障检测、领导者选举和自动故障转移三大能力来解决Redis主从架构的单点故障问题。在原理层面，哨兵通过主观下线和客观下线两道检测关卡确保故障判断的准确性，采用类似Raft的领导者选举机制避免多哨兵同时执行故障转移造成的脑裂问题。配置文件中每个参数都直接影响哨兵的行为：down-after-milliseconds控制故障检测的灵敏度，quorum控制触发故障转移的共识阈值，parallel-syncs控制数据同步的压力。

在实际部署中，哨兵集群应该以奇数节点部署在独立于Redis数据节点的服务器上，并通过动态配置和日志分析来确保持续可用的监控能力。故障转移完成后的数据一致性验证是不可跳过的步骤，而结合Prometheus和Grafana的监控告警体系则让运维工作从被动响应转向主动预防。

掌握哨兵机制是构建Redis高可用架构的基础，但它并非万能药。对于更大规模的数据分片需求，需要引入Redis Cluster；对于需要跨机房容灾的场景，则需要在哨兵之上叠加更复杂的网络层和流量调度方案。理解每个方案的适用边界，才能在实际项目中做出正确的技术选型。

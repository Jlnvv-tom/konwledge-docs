# 第11章 Redis消息队列实现与应用

很多人对 Redis 的第一印象还停留在“缓存”，但只要你的系统里出现了“异步解耦”“削峰填谷”“任务分发”这些词，消息队列就会自然而然地摆上台面。这时候一个很现实的问题浮出来：是直接上 Kafka、RabbitMQ 这类专业 MQ，还是先用手边已有的 Redis 顶一顶？我在好几个项目里都做过这个选择，也踩过对应的坑——用 List 做队列丢过消息，用 Pub/Sub 做通知漏过订单，最后切到 Stream 才把可靠性这件事真正立起来。

这一章我不打算把 Redis 消息队列吹成万能药，而是老老实实把三种实现方式（List、Pub/Sub、Stream）从原理到代码摆开，把每一种的适用边界和翻车点标清楚。读完你应该能回答两个问题：Redis 做消息队列到底靠不靠谱？在什么规模、什么可靠性要求下，它够用，什么时候必须换专业 MQ。

我想先给一个总的态度：Redis 做消息队列，是一种“够用主义”的选择。它的优势从来不是功能有多强，而是“你本来就有它”。团队里已经跑着 Redis，运维熟、监控现成、依赖不用加，这种“零边际成本”的诱惑在中小项目里几乎无法拒绝。很多所谓“上了 Kafka 才后悔”的故事，根源不是 Kafka 不好，而是当时那点消息量根本不配拥有一套 Kafka 集群的运维负担。反过来，也有团队用 Redis List 硬扛核心交易消息，最后被丢消息坑到深夜复盘。所以这一章的真正价值，不是教你几个命令，而是帮你在“够用”和“过度设计”之间找到那条线。

## 11.1 消息队列的核心概念：生产者、消费者与消息可靠性

在动手写代码之前，得先把几个概念对齐，否则后面讨论“消息会不会丢”时容易鸡同鸭讲。

消息队列本质上是一个中间的缓冲区。生产者（Producer）负责往里放消息，消费者（Consumer）负责从里面取消息处理，两者不直接通信，靠这个队列解耦。带来的好处是显而易见的：生产端不用等消费端处理完，请求打进来先落队列就返回，响应飞快；消费端按自己的节奏慢慢消化，流量高峰时队列帮你把洪峰压平，这就是所谓的削峰填谷。此外还能异步化耗时操作，比如下单后发短信、发优惠券这类非核心链路，全部丢队列异步做，主链路只管下单。

但真正决定一个消息队列能不能用在生产上的，是消息可靠性。这里要拆成三个层面来看，每一层都对应着一类线上事故。

第一层是消息不丢，也就是投递可靠性。消息从生产者发出，到最终被消费者成功处理，中间任何一个环节（网络、Broker 宕机、消费者崩溃）都不能让消息凭空消失。这一层又细分为三个子问题：生产者发出去了 Broker 有没有收到、Broker 存下来了会不会因为重启丢、消费者拿到了处理到一半崩了消息还能不能被重新处理。这三个子问题对应三道防线：生产端要有发送确认（知道 Broker 收到了），Broker 要有持久化（落盘不丢），消费端要有消费确认（处理完才确认）。任何一道缺失都会漏消息，后面你会看到 Redis 三种方案恰恰就是在这三道防线上逐步补齐的：List 有持久化但缺消费确认，Pub/Sub 三道都缺，Stream 三道都有。

第二层是消息不重复，或者更准确地说，是消费幂等。为了保证不丢，很多机制会采用“至少投递一次”（at-least-once）策略，代价就是可能重复投递。所以消费端必须自己做幂等——同一条消息处理一次和处理十次，结果要一致。这个锅不能全甩给队列，业务层必须兜底。

第三层是消息有序。有些场景（比如同一个订单的“创建、支付、发货”状态流转）要求消息按发送顺序被消费，一旦乱序就会出现“先发货后支付”这种灵异事件。需要注意的是，有序往往和高并发互相矛盾：你一旦开多个消费者并行处理来提吞吐，就很难保证全局有序。所以业界的做法都是“局部有序”——只保证同一个业务实体（如同一订单 ID）的消息有序，不同实体之间不保证。这个思路在后面讲 Stream 分流时还会再提。

这里我想用一个真实场景把“削峰填谷”讲透，因为很多人只背下了名词。我们做过一个秒杀活动，瞬时涌入几万笔下单请求，如果每笔请求都同步去写数据库、扣库存、生成订单、发短信，数据库直接被打穿。后来的做法是，前置链路只做库存预扣（Redis 原子操作）和“下单请求入队”两件事，成功了就立即给用户返回“抢到了，正在处理”；真正的创建订单、扣减数据库库存都扔给后面的消费者，按数据库能承受的速度慢慢消化。用户体验上“瞬间抢到”，实际上订单可能是几秒后才真正落库的。这就是队列的价值：把瞬时洪峰用队列接住，后端按自己的节奏处理，不被峰值冲垮。

解耦的价值同样值得展开。还是下单这件事，早期我们把“下单后发优惠券、加积分、推送消息、通知商家”全写在一个同步方法里，结果积分服务一抖，整个下单接口跟着超时，用户下不了单。明明积分是个非核心的旁支功能，却把核心链路拖下水。改成消息队列后，下单主链路只管创建订单，其他旁支全部丢队列异步做，即使积分服务挂了，也不影响下单，大不了积分晚到几秒。这种“核心链路短、旁支异步化”的思路，是高可用系统设计的基本功。

把这三层记牢，后面评价 List、Pub/Sub、Stream 就有了统一标尺。可以先剧透结论：Pub/Sub 三层几乎全不满足，List 能勉强满足“不丢”的一部分，Stream 才是三层都有像样支持的那个。理解了这个框架，你就不会被“Redis 能不能做消息队列”这种笼统的问题绕进去——答案永远是“看你要哪个层面的可靠性”。

顺带说一个认知误区。很多人觉得“用了消息队列就一定可靠”，其实恰恰相反，引入 MQ 反而增加了系统的复杂度和失败面：多了一个需要维护的中间件，多了消息丢失、重复、堆积、顺序错乱这些新问题。所以选型时别只看“能不能实现功能”，更要看“为了可靠性你愿意付出多少工程成本”。Redis 的价值就在于，它能在中等可靠性要求下，用极低的引入成本满足你——你本来就部署了 Redis，不需要再运维一套 Kafka。

再补一个实战里经常被忽略的维度：投递语义的三种档位。一是“至多一次”（at-most-once），发了就不管，可能丢，但绝不重复；二是“至少一次”（at-least-once），保证不丢，但可能重复；三是“恰好一次”（exactly-once），不丢不重，这是圣杯。要特别清醒地认识到：包括 Kafka 在内的绝大多数中间件，默认给的都是“至少一次”，所谓的 exactly-once 大多是靠下游幂等性拼出来的，而不是中间件魔法般保证的。换句话说，无论你用 Redis 还是 Kafka，只要想要不丢，就必然要在业务层自己做幂等。这个认知立住了，后面看 Stream 的 XACK 机制就不会有不切实际的期待。

还有一个和团队协作相关的软性问题：消息队列一旦引入，就把“同步调用栈”拆成了“异步事件流”，排查问题的方式完全变了。以前一个请求出错，翻一条调用链就能定位；用了 MQ，一条业务可能被切成生产、入队、消费、重试好几段，散落在不同服务、不同时间点的日志里。所以我强烈建议：任何消息里都带上一个全局唯一的 traceId，生产时打日志、消费时打日志，出问题能顺着 traceId 把整条链路串起来。这个习惯在你第一次半夜排查“消息去哪了”时，会救你的命。

## 11.2 基于List的消息队列实现：lpush与brpop组合

Redis 的 List 是一个双向链表，天然支持从两端进出。用它做消息队列的思路极其朴素：生产者用 `LPUSH` 从左边塞消息，消费者用 `RPOP` 从右边取消息，一进一出正好构成一个先进先出（FIFO）的队列。当然你也可以反过来用 `RPUSH` + `LPOP`，只要生产和消费从相反的两端操作就是 FIFO，从同一端操作就变成了后进先出的栈。这个方向一定要在团队里约定好，我见过生产者和消费者代码由不同人写，一个用 LPUSH、一个也用 LPOP，结果变成了栈，老消息永远压在底下没人处理。

先看最基础的写法：

```bash
# 生产者：往队列左侧推入消息
LPUSH order:queue "{\"orderId\":1001,\"action\":\"pay\"}"

# 消费者：从队列右侧弹出消息
RPOP order:queue
```

这个方案的问题马上就来了：`RPOP` 是非阻塞的，队列为空时它直接返回 nil。那消费者怎么知道有没有新消息？最蠢的办法是写个死循环不停 `RPOP`，也就是忙轮询（busy polling）。我早期真这么干过，结果 CPU 空转、Redis QPS 被无意义的空查询打满，监控上一片红。

正确的姿势是用阻塞版本 `BRPOP`。队列为空时它会阻塞挂起，直到有新消息推入或超时才返回，完全不占 CPU：

```java
// 消费者：阻塞式获取，超时时间 0 表示永久阻塞
public void consume() {
    while (running) {
        // BRPOP 返回 [key, value]，超时返回 null
        List<String> result = redisTemplate.opsForList()
            .rightPop("order:queue", 30, TimeUnit.SECONDS);
        if (result != null) {
            handle(result.get(1));
        }
    }
}
```

这里有个细节：我不建议把超时设成 0（永久阻塞）。生产环境里如果 Redis 发生主从切换、连接被断开，永久阻塞的客户端可能感知不到，卡死在那里。设一个 30 秒左右的超时，让它周期性醒来一次，既能借机检查连接健康度，也方便优雅关闭时跳出循环。这是我在一次哨兵切换后消费者集体假死的事故里学到的教训——所有消费者都还“连着”，但一条消息都不消费了，因为它们阻塞在一条已经失效的连接上。

List 队列的优点是简单、有序（严格 FIFO）、消息持久化（存在 Redis 里，重启不丢，前提是开了 RDB/AOF）。但它的坑同样明显，逐个说。

这里顺带解释一下 `BRPOPLPUSH` 和 6.2+ 新增的 `BLMOVE` 的关系：前者只能左推右弹（方向固定），后者可以自由指定从哪端取、放到目标队列的哪端，更灵活。Redis 6.2 开始官方把 `BRPOPLPUSH` 标为不推荐（deprecated），建议用 `BLMOVE` 替代，新项目直接用 `BLMOVE` 就好。两者底层都是原子的“弹出并转存”，不会出现“弹出了但还没来得及转存就崩溃”的中间态，这才是它们能用来做可靠队列的关键。

第一个坑：消息确认缺失导致丢消息。`BRPOP` 一执行，消息就从队列里被删掉了。如果消费者刚弹出消息，还没处理完就崩溃了，这条消息就永远消失了，没有任何补救机会。这是 List 方案最致命的短板——它没有 ACK 机制。要缓解，得用 `BRPOPLPUSH`（或 6.2+ 的 `BLMOVE`），把消息从主队列原子地转移到一个“处理中”备份队列，处理成功后再从备份队列删除；如果消费者崩了，可以有个补偿任务扫描备份队列把超时未处理的消息捞回来重投。

```java
// 可靠消费：弹出的同时备份到 processing 队列
String msg = redisTemplate.opsForList()
    .move("order:queue", RIGHT, "order:processing", LEFT,
          Duration.ofSeconds(30));
if (msg != null) {
    try {
        handle(msg);
        // 处理成功后从 processing 中移除该消息
        redisTemplate.opsForList().remove("order:processing", 1, msg);
    } catch (Exception e) {
        // 不删除，留给补偿任务重投
    }
}
```

这里多说一句补偿任务的设计。processing 备份队列里的每条消息，最好带上一个“开始处理时间戳”。补偿任务定时扫描这个队列，把那些处理时间超过阈值（比如 5 分钟）还没被删除的消息重新推回主队列。但要特别小心一个并发陷阱：补偿任务把消息重推后，原消费者又恢复了把它处理完，就会重复处理，所以这套方案同样逃不开“至少一次”的幂等要求。看到没，不管你怎么用 List 往可靠靠，最后都会回到“业务层幂等”这个铁则上。

第二个坑：无法多消费者组独立消费。一条消息被一个消费者 `BRPOP` 走了，别的消费者就拿不到了。这在“一条消息要被多个不同业务系统各自消费一遍”的广播场景下完全没法用——你想让订单系统、积分系统、推送系统都收到同一条“下单成功”消息，List 做不到，它天生是“抢占式”的点对点模型。

第三个坑：无消息回溯。消息一旦被消费就删除，没法重新读取历史消息。如果某个消费者逻辑有 bug，处理错了想重放一批消息，对不起，消息早没了。

第四个坑：堆积监控要自己做。队列长度得靠 `LLEN` 主动去查，没有内建的堆积告警。消息大量堆积时，还会撑爆 Redis 内存，因为 List 里的消息全在内存里。我们的做法是用一个定时任务每分钟采一次 `LLEN`，超过阈值就告警，否则一旦消费端挂了，堆积会在几分钟内把 Redis 内存吃光，连带把共用这台 Redis 的其他业务一起拖垮。所以用 List 做队列，强烈建议和缓存业务物理隔离，别混用同一实例。

第五个坑：消息体只能是字符串，序列化得自己管。List 里存的就是一个个字符串，你得自己把对象序列化成 JSON（或者更紧凑的二进制）再推进去，消费端再反序列化回来。这本身不难，但易错的是版本兼容：生产者改了消息结构、加了字段，而消费者还是旧版本，反序列化直接报错。我们的做法是消息体里带一个 version 字段，消费端根据版本做兼容解析，并且新增字段只加不改、不删旧字段，保持向后兼容。这个约束在任何消息队列里都适用，不只是 List。

所以 List 方案适合什么？适合那种“单一消费者组、点对点、消息量不大、丢一两条能忍或有补偿机制”的轻量异步任务，比如内部的异步任务分发。真要追求可靠性和多消费组，它就力不从心了。

顺带提一个很多人想拿 List 做的需求：延时队列。比如“下单 30 分钟未支付自动取消”。单纯用 List 做不了延时，因为它只能 FIFO 顺序消费。业界常见的做法是改用 ZSet（有序集合），把“应执行时间戳”作为 score，消费者用 `ZRANGEBYSCORE` 轮询把到期的消息捞出来。不过这种轮询有惊群和并发重取问题，需要配合 Lua 脚本把“取出 + 删除”做成原子操作。好消息是，Stream 出来后，延时队列也有了更好的承载方式（配合外部调度），但真正重延时、重可靠的场景，我依旧推荐专业 MQ 的延时消息特性（如 RocketMQ 的延级消息）。

## 11.3 基于Pub/Sub的消息订阅与发布

如果说 List 是点对点的“抢消息”，那 Pub/Sub（发布/订阅）就是广播式的“喊话”。生产者往一个频道（channel）`PUBLISH` 消息，所有 `SUBSCRIBE` 了这个频道的消费者都会同时收到一份。这正好补上了 List 无法多消费者广播的短板。

基本用法非常直观：

```bash
# 订阅者 A 和 B 都订阅 news 频道
SUBSCRIBE news

# 发布者往 news 频道发消息，A 和 B 同时收到
PUBLISH news "hello everyone"
```

这里要强调一个和 List 本质不同的点：List 是“抢占式”，一条消息只能被一个消费者拿走；而 Pub/Sub 是“广播式”，一条消息会同时发给所有订阅者。这两种模型没有优劣，只有适用场景不同：任务分发用 List（一个任务只能被处理一次），事件广播用 Pub/Sub（一个事件让所有关心者都知道）。搞混这两个模型，是新手设计消息链路时最常犯的方向性错误。

除了精确匹配的 `SUBSCRIBE`，Redis 还支持模式订阅 `PSUBSCRIBE`，可以用通配符一次订阅一批频道，这在做“按业务前缀分频道”的场景里很方便：

```bash
# 订阅所有 order. 开头的频道
PSUBSCRIBE order.*

# 下面两条消息，上面的订阅者都能收到
PUBLISH order.create "{...}"
PUBLISH order.pay "{...}"
```

Java 侧用 Spring Data Redis 集成也很简单，注册一个消息监听容器即可。这里有个实战注意点：监听容器内部用的是一个独立的线程池回调 onMessage，如果你在回调里干了耗时活（比如同步调下游接口），线程池被打满后后续消息就处理不过来，叠加上 Pub/Sub 本身不缓存的特性，很容易丢消息。所以回调里只做轻量活，重活丢线程池异步处理，并且别忘了回调里做异常捕获，否则一个未捕获异常可能把监听线程弄挂：

```java
@Bean
public RedisMessageListenerContainer container(
        RedisConnectionFactory factory) {
    RedisMessageListenerContainer c = new RedisMessageListenerContainer();
    c.setConnectionFactory(factory);
    // 监听 news 频道，收到消息回调 onMessage
    c.addMessageListener((message, pattern) -> {
        System.out.println("收到: " + new String(message.getBody()));
    }, new ChannelTopic("news"));
    return c;
}
```

Pub/Sub 真正的用武之地不是“消息队列”，而是“实时通知”。它的模型是“发出去有人听就听，没人听就算了”，特别适合那些对丢失不敏感、只关心实时性的场景。举几个我实际用过的：

一是配置热更新。当管理后台修改了某个配置，往一个 `config:update` 频道 `PUBLISH` 一下，所有应用节点订阅这个频道，收到后重新拉取配置，实现无重启的配置下发。Nacos、Apollo 这类配置中心的底层通知机制思路上很类似。这里有个小技巧：广播的不是配置内容本身，而只是一个“配置变了，去拉”的信号。因为 Pub/Sub 可能丢消息，如果直接把配置值广播出去，某个节点恰好没收到就会长期用旧值；而只广播信号、节点主动去拉权威数据源，即使丢了一次信号，下次重启或定时全量拉取时也能纠正。把 Pub/Sub 当“触发器”而不是“搬运工”，是用好它的关键思路。

二是缓存失效广播。多节点本地缓存（如 Caffeine）要保持一致时，某个节点更新了数据，就往 `cache:invalidate` 频道广播一条 key，所有节点收到后清掉自己本地对应的缓存。

三是 Redis 自带的 Keyspace Notification，本质就是 Redis 内部用 Pub/Sub 把 key 的过期、删除等事件推给订阅者。上一章讲 Session 过期清理时用到的就是它。顺带提醒，正因为 Keyspace Notification 底层是 Pub/Sub，它同样会丢事件，所以千万别拿它做“精确的延时任务触发”——订阅者重启那几秒过期的 key 事件就丢了，任务就不会触发。很多人拿它做延时队列翻过车，根源就在这里。

四是聊天室、弹幕、在线推送这类强实时、弱可靠的场景。用户刷直播发个弹幕，往房间频道一 PUBLISH，房间里其他人实时看到。弹幕偶尔丢一条根本无所谓，但要求延迟极低，这恰恰是 Pub/Sub 的主场。记住一个判断口诀：“实时优先、丢了不心疼”的选 Pub/Sub，“一条都不能丢”的选 Stream。

Pub/Sub 看起来很美，但它有一个根本性的致命缺陷，这也是 11.4 要专门展开的话题。这里先埋个伏笔：它压根不是一个“队列”，因为它不存储消息。

## 11.4 Pub/Sub的局限性：无持久化与消息丢失问题

我第一次栽在 Pub/Sub 上，是拿它做了订单状态的异步通知。逻辑很简单：订单支付成功后 `PUBLISH` 一条消息，下游的积分服务订阅后给用户加积分。测试环境跑得好好的，上线后陆续有用户反馈支付了没到账积分。查了半天日志，发现问题出在积分服务重启的那几秒——重启期间发的消息全丢了。

这就是 Pub/Sub 最核心的局限：无持久化，消息发完即焚。`PUBLISH` 的语义是“把消息推给当前在线的订阅者”，如果某个订阅者此刻不在线（宕机、重启、网络抖动断连），那它就永远收不到这条消息了。消息不会被存下来等你回来再取，Redis 推完就把它丢了。这套模型叫 fire-and-forget，是典型的“至多投递一次”（at-most-once），换句话说，它默认就允许丢消息。

把 Pub/Sub 的坑逐条列清楚，你就明白为什么它不能当可靠队列用：

第一，消费者离线期间的消息全部丢失。这是最要命的一条。任何一次消费者重启、发布、扩缩容、网络闪断，都会造成窗口期内的消息黑洞。而生产系统里，消费者重启是家常便饭。

第二，没有 ACK 机制。消息推给你就算完事，Redis 不关心你有没有真正处理成功。消费者收到消息后处理到一半崩了，这条消息也没了，没有重试。

第三，消息堆积会被丢弃。如果消费者处理速度跟不上发布速度，消息会积压在消费者的输出缓冲区里。Redis 对 Pub/Sub 客户端有 `client-output-buffer-limit` 限制，超过阈值 Redis 会直接把这个慢消费者的连接断开，缓冲区里没来得及发的消息统统丢弃。这意味着慢消费者不仅丢消息，还会被踢下线。更阴险的是这个限制默认值对 pubsub 客户端是有限的（例如 32mb / 8mb 60），很多人根本不知道有这个配置，直到消费者莫名其妙被断连、日志里冒出一堆重连才去查。而且它和 List/Stream 完全不同：List 堆积不过是占内存，消息还在；Pub/Sub 堆积到阈值就是直接丢掉，无可挡回。

第四，无法回溯历史。和 List 一样，发过的消息读不回来，没有任何历史记录。

第五，跨集群分片问题。在 Redis Cluster 下，普通 Pub/Sub 的消息要在所有节点间广播（消息不分片），大规模集群里这会带来额外的网络开销。Redis 7.0 引入了 `SPUBLISH`/`SSUBSCRIBE`（分片 Pub/Sub）来缓解，但也就限制了广播范围。

所以结论必须说死：Pub/Sub 绝对不能用于任何要求“消息不能丢”的业务场景。订单、支付、积分、库存这类，一条都不能少的，碰都别碰 Pub/Sub。它只适合那些“丢了也无所谓、我只要实时”的通知类场景——配置推送、状态广播、聊天室在线消息这种。判断标准很简单:问自己一句“这条消息如果因为对方刚好重启而丢了，业务能接受吗？”能接受才用 Pub/Sub，不能接受就往下看 Stream。

还有一个很隐蔽的坑：订阅连接断开后的重连真空期。客户端一旦发生网络抖动、连接断开，到它重新 SUBSCRIBE 成功之前的这段时间，所有发布的消息都收不到。即使你用了自动重连的客户端，重连也不能把断开期间的消息补回来。我那次积分事故其实不只是重启，后来复盘发现那阵子网络抖动导致的短暂断连也同样丢了一批，只是量少没被立即发现。这种“阴魂不散”的零星丢失，比一次性大量丢失更难排查，因为它不会触发告警，只能靠事后对账发现。

有人会问，那我给 Pub/Sub 加个持久化补丁行不行？比如发布的同时也往 List 里存一份。这其实是在自己造轮子，而且造得四不像——你要自己维护偏移量、自己做 ACK、自己处理重复。既然如此，为什么不直接用官方已经把这些都做好的 Stream 呢？这就引出了本章的主角。

## 11.5 Redis Stream：Redis 5.0+的持久化消息队列

Redis 5.0 推出的 Stream，是官方专门为“消息队列”这个场景设计的数据结构。它吸收了 Kafka 的很多设计理念，把前面 List 和 Pub/Sub 的短板一次性补齐了。可以说，从 Stream 开始，Redis 才真正拥有了一个“像样”的消息队列。事实上，Stream 的作者 antirez 当年就是因为看到大量用户在拿 List 和 Pub/Sub 硬凑消息队列、又被各种丢消息问题困扰，才决定做一个专门的结构把这些痛点一次解决。所以你会发现 Stream 的每一个特性——持久化、消费组、ACK、回溯——都精准地对应着前两者的一个痛点，它不是凭空设计的，而是从实际翻车中长出来的。

先看 Stream 到底解决了什么，把它和前两者对比一下就一目了然。

消息持久化方面，Stream 是一个只追加（append-only）的日志结构，消息发进去就存下来，随 RDB/AOF 落盘，重启不丢。这解决了 Pub/Sub 发完即焚的问题。

消息回溯方面，每条消息都有一个唯一且单调递增的 ID（默认是“毫秒时间戳-序号”格式，如 `1699000000000-0`），消费者可以按 ID 从任意位置重新读取历史消息。这解决了 List 和 Pub/Sub 都没有的历史回溯问题。

多消费者组方面，Stream 引入了消费组（Consumer Group）概念，这是它最强大的特性。同一个 Stream 可以有多个消费组，组与组之间各自独立地消费全量消息（广播语义，像 Pub/Sub）；而同一个组内的多个消费者则分摊消费（负载均衡，像 List 的点对点）。两种模型合二为一，既能广播又能分流。

消息确认方面，Stream 有完整的 ACK 机制。消费组读取消息后，消息进入该消费者的“待处理列表”（Pending Entries List，简称 PEL），只有消费者显式 `XACK` 确认后，消息才算处理完成。如果消费者崩了没确认，消息还留在 PEL 里，可以被重新认领处理。这就实现了“至少一次”投递，彻底解决了丢消息问题。

把四种能力和前面的方案排一起对比：List 有持久化、有序，但无回溯、无消费组、无 ACK；Pub/Sub 有广播，但无持久化、无回溯、无 ACK、无堆积保护；Stream 则是持久化、可回溯、多消费组、有 ACK 全都有。从功能完整度看，Stream 是 Redis 里唯一能称得上“可靠消息队列”的选项。

这里需要理解一下消费组两种语义合一的威力。举个例子：一条“下单成功”消息，你想让积分系统、推送系统、数据仓各自消费一遍（广播），同时每个系统内部又想开多个实例分摊压力（负载均衡）。用 Stream 就是：给积分、推送、数据仓各建一个消费组，三个组都能收到全量消息；而积分组内部开 consumer-1、consumer-2 两个消费者，一条消息只会发给其中一个。广播到组、分流到人，一套机制同时满足 Pub/Sub 和 List 两种需求，这是前两者都做不到的。熟悉 Kafka 的人会发现，这就是 Kafka 的 consumer group 模型，Stream 确实大量借鉴了 Kafka 的设计。

当然，Stream 也不是没有代价。它的内存占用比 List 大（每条消息带 ID、每个消费组维护 PEL），操作也更复杂（命令多、心智负担重）。而且它终究是内存数据库上的队列，在吞吐量、消息堆积能力（TB 级持久化）、生态工具（监控、死信、事务消息）上，和 Kafka、RocketMQ 这类专业 MQ 还是有明显差距。所以 Stream 的定位很清晰：中小规模、已有 Redis、不想额外引入重型 MQ 的可靠消息场景。日均消息量在千万级以内、不需要海量堆积和复杂治理的系统，Stream 完全够用；一旦到了日均数亿、需要长期存储和复杂路由的量级，还是老老实实上专业 MQ。

这里有一个很多人会混淆的点，值得特别澄清：Stream 的“消费”并不会把消息从 Stream 里删掉。你 `XREADGROUP` 读走一条消息、甚至 `XACK` 确认了，消息本体依旧躺在 Stream 这个日志里。XACK 只是把它从“待处理列表”（PEL）里移除，表示“这个组处理完了”，但消息实体还在，只能靠 `XADD` 时的 `MAXLEN` 或 `XTRIM` 命令才能真正删掉。这和 List“弹出即删除”的模型完全不同，也正是 Stream 能支持多消费组各自消费全量、能回滯历史的根本原因——数据只有一份，每个组各自用一个“游标”记录自己读到哪了。

理解了定位，下一节我们就把 Stream 的核心命令一个个敲一遍，重点放在消费组的实战用法上，因为那才是 Stream 的精华所在。

## 11.6 Stream核心操作：xadd、xread、xgroup等与消费组实现

这一节是全章最硬核的部分。我按“生产 → 简单消费 → 消费组消费 → 异常处理”的顺序，把 Stream 的核心命令串起来讲，每一步都配上代码和踩坑点。

先是生产消息，用 `XADD`。它往 Stream 追加一条消息，消息体是一组 field-value 键值对：

```bash
# * 表示让 Redis 自动生成消息 ID
XADD order:stream * orderId 1001 action pay amount 99
# 返回生成的消息 ID，如 1699000000000-0
```

生产上有个必须注意的点：Stream 默认会无限增长，消息只进不出会撑爆内存。所以 `XADD` 时应该带上 `MAXLEN` 做长度控制，配合 `~` 做近似裁剪（性能更好）：

```bash
# 保留大约最近 10000 条，~ 表示近似裁剪，性能更优
XADD order:stream MAXLEN ~ 10000 * orderId 1001 action pay
```

`MAXLEN ~ 10000` 的意思是“大约保留 1 万条，多出来的老消息会被裁掉”。这里的 `~` 很关键，加了它 Redis 可以按节点整块裁剪而不是精确到条，性能高很多。这是我在一次 Stream 内存暴涨事故后加上的——当时没设 MAXLEN，一个高频 Stream 几天就吃掉几个 G 内存。不过要注意一个潜在风险：裁剪只看消息总数，不看“是否已被消费”。如果某个消费组卡住、消费得比生产慢很多，MAXLEN 可能把它还没来得及读的消息也裁掉，造成丢消息。所以 MAXLEN 的阈值要留足冗余，并且一定要配合对消费堆积（lag）的监控，确保裁剪线远远落在最慢消费组的进度后面。真需要按时间而不是条数裁，可以用 `MINID` 参数把早于某个 ID（时间戳）的消息裁掉。

顺带把 Stream 的消息 ID 讲清楚，因为很多操作都围绕它。ID 默认格式是 `毫秒时间戳-序号`，比如 `1699000000000-0`，同一毫秒内多条消息靠后面的序号区分。这个设计保证了 ID 全局单调递增，也就天然支持按时间回溯：想读某个时间点之后的消息，把时间戳拼成 ID 丢给 `XRANGE` 就行。你也可以 `XADD` 时自己指定 ID，但必须比现有最大 ID 大，否则报错，这个约束保证了日志的追加有序性。

最简单的消费方式是 `XREAD`，它不需要消费组，直接按 ID 读消息，适合单消费者场景：

```bash
# 从 ID 0 开始读，COUNT 限制条数，BLOCK 阻塞等待新消息
XREAD COUNT 10 BLOCK 5000 STREAMS order:stream 0

# 特殊 ID $ 表示只读此刻之后的新消息（类似 Pub/Sub）
XREAD BLOCK 0 STREAMS order:stream $
```

但 `XREAD` 只是个“花哨的 List 消费”，没有消费组、没有 ACK，多个消费者一起读会各读各的全量（没有分流），而且它不会帮你记录“读到哪了”，偏移量得你自己在客户端维护（把每次读到的最大 ID 存下来，下次从它之后读）。一旦客户端重启、偏移量丢了，要么重复消费、要么漏消费。这也是为什么 `XREAD` 只适合单机工具、监控面板这类“丢了无所谓”的读取，真正的生产用法是消费组——它把偏移量和 PEL 都存在 Redis 服务端，客户端无状态，重启也不会乱。这才是 Stream 的灵魂。

消费组的核心命令是 `XGROUP`（管理组）、`XREADGROUP`（组内消费）、`XACK`（确认）。先创建消费组：

```bash
# 为 order:stream 创建消费组 order-group
# 0 表示从头开始消费，$ 表示从最新开始消费
XGROUP CREATE order:stream order-group 0 MKSTREAM
```

`MKSTREAM` 参数很实用，如果 Stream 还不存在，它会顺便把 Stream 建出来，避免“组创建失败因为 Stream 不存在”的报错。这是新手最常踩的第一个坑——先建组还是先建流，用 `MKSTREAM` 一步到位。另一个常见报错是重复创建同名消费组会报 BUSYGROUP，所以应用启动时创建消费组的代码要把这个错误当成“已存在，正常”处理掉，而不是直接抛出来阻断启动。我们的做法是封装一个 ensureGroup 方法， try 住创建异常，发现是 BUSYGROUP 就忽略，其他异常才向上抛，这样服务反复重启都不会有问题。

这里再提一个新手容易困惑的点：创建消费组时的起始位置选 0 还是 $。选 0 表示从 Stream 最头开始消费，会把建组之前已经存在的历史消息全部重新消费一遍；选 $ 表示只消费建组之后新来的消息，旧消息全部忽略。这个选错会出大事：本想消费新消息却用了 0，服务一启动就把历史几十万条消息全重跑了一遍；反过来本想从头补消费却用了 $，结果旧消息永远不会被处理。一般业务新建组用 $，需要回放或首次全量处理才用 0。

然后组内消费者用 `XREADGROUP` 读消息。同一个组里，每条消息只会分给其中一个消费者，实现负载均衡：

```java
// consumer-1 从消费组读消息，> 表示读取从未投递过的新消息
public void consumeGroup() {
    while (running) {
        List<MapRecord<String, Object, Object>> records =
            redisTemplate.opsForStream().read(
                Consumer.from("order-group", "consumer-1"),
                StreamReadOptions.empty().count(10).block(Duration.ofSeconds(5)),
                StreamOffset.create("order:stream", ReadOffset.lastConsumed()));
        for (MapRecord<String, Object, Object> record : records) {
            try {
                handle(record.getValue());
                // 处理成功后确认，消息从 PEL 移除
                redisTemplate.opsForStream().acknowledge("order-group", record);
            } catch (Exception e) {
                // 不 ACK，消息留在 PEL 待重试
            }
        }
    }
}
```

这里的 `ReadOffset.lastConsumed()` 对应命令里的 `>`，表示“读还没投递给本组任何消费者的新消息”。要区分的是，如果把 `>` 换成一个具体 ID（比如 0），`XREADGROUP` 读的就不是新消息，而是该消费者 PEL 里那些已经投递但还没 ACK 的消息。这个特性很有用：消费者重启后，应该先用 `0` 把自己 PEL 里没处理完的消息重新拉出来处理一遍（因为它们已经分给了这个消费者，不会自动转给别人），处理完再切到 `>` 消费新消息。不做这一步的话，这些消息会一直卡在 PEL 里等着补偿线程用 XCLAIM 来捞，处理延迟变大。读到后消息进入这个消费者的 PEL，等你 `XACK` 确认才移除。如果你不 ACK，消息会一直躺在 PEL 里——这既是可靠性的来源，也是新手第二个大坑：忘了 ACK，导致 PEL 无限膨胀，看起来消息消费了但 Redis 认为你没处理完，内存越涨越高。我 code review 时最常揪出来的就是漏 `XACK`。

现在来看异常处理，这才是消费组真正体现价值的地方。假设 consumer-1 读了一条消息，还没 ACK 就崩溃了。这条消息卡在 consumer-1 的 PEL 里，不会被自动重投给别人。怎么办？靠 `XPENDING` 和 `XCLAIM`。

`XPENDING` 用来查看某个组里有哪些消息处于待确认状态、卡在谁手里、卡了多久：

```bash
# 查看 order-group 组的 pending 概况
XPENDING order:stream order-group
# 查看详情：卡了超过 60000ms 的消息，最多 10 条
XPENDING order:stream order-group IDLE 60000 - + 10
```

发现有消息在崩溃的 consumer-1 那里卡了很久，就用 `XCLAIM` 把它“认领”到一个健康的消费者名下，让后者重新处理。这里的“认领”其实就是把消息从一个消费者的 PEL 转移到另一个消费者的 PEL，同时重置它的 idle 时间。消息本体一直在 Stream 里，没有任何拷贝，只是“归属权”变了：

```bash
# 把卡了超过 60s 的消息转移给 consumer-2 重新处理
XCLAIM order:stream order-group consumer-2 60000 1699000000000-0
```

Redis 6.2 还提供了更省事的 `XAUTOCLAIM`，能自动扫描并批量认领超时消息，不用你先 `XPENDING` 再挨个 `XCLAIM`：

```bash
# 自动认领 idle 超过 60s 的消息给 consumer-2，一次最多 10 条
XAUTOCLAIM order:stream order-group consumer-2 60000 0 COUNT 10
```

生产上，你通常会起一个后台线程定时跑 `XAUTOCLAIM`，把那些长期卡在 PEL 里的消息捞出来重投，这就是 Stream 的故障转移能力。这套“PEL + XCLAIM”机制，正是 Stream 相比 List 最本质的可靠性优势——消费者崩了，消息不丢，还能被别人接手。这里有个细节要提醒：idle 超时阈值（上面的 60000ms）不能设得太短。如果一条消息正常处理就要 30 秒，你把 idle 阈值设成 10 秒，补偿线程会把还在正常处理中的消息误认为“卡住”而抢走重投，造成不必要的重复消费。阈值应该明显大于一条消息的正常处理耗时，留足余地。同时，补偿线程自己也要防高频空跑，建议隔几秒扫一次就够了，别写成死循环。

再补两个实战要点。第一是死信处理。如果一条消息反复处理失败、`XCLAIM` 认领次数（delivery count，`XPENDING` 里能看到）超过阈值（比如 3 次），说明它可能是条“毒消息”，别再无限重试了，把它转移到一个专门的死信 Stream 单独排查，同时 `XACK` 掉原消息避免它继续堵塞。Redis 没有内建死信队列，这个逻辑得自己实现：

```java
// deliveryCount 来自 XPENDING 返回，超过阈值则转死信
if (deliveryCount > 3) {
    // 把消息原样写入死信 Stream，保留现场
    redisTemplate.opsForStream().add("order:stream:dlq",
        record.getValue());
    // 确认原消息，让它不再堵塞正常消费
redisTemplate.opsForStream().acknowledge("order-group", record);
    return;
}
```

死信队列不是终点，而是一个“人工兵”入口。我们会给死信 Stream 配上告警，一旦有消息落进去就通知值班，人工看一眼到底是数据脏了、还是下游服务长期挂了，修复后可以手动把死信消息重新注入正常 Stream 重试。没有死信队列的后果是，一条毒消息会卡在 PEL 里无限重试，既浪费资源又可能阻塞后面的正常消息。

第二是消费幂等。前面反复强调过，Stream 是“至少一次”投递，`XCLAIM` 重投必然带来重复消费。所以消费端一定要做幂等，最常见的做法是拿消息里的业务唯一键（如 orderId）去 Redis 或数据库里查一个“已处理”标记，处理过就直接 `XACK` 跳过。这个兜底不能省，否则用户可能被加两次积分、扣两次库存。一个轻量的幂等实现如下：

```java
// 用 SETNX 做幂等标记，已处理则直接跳过
String idemKey = "idem:order:" + orderId;
Boolean first = redisTemplate.opsForValue()
    .setIfAbsent(idemKey, "1", Duration.ofHours(24));
if (Boolean.FALSE.equals(first)) {
    return; // 已处理过，直接忽略
}
try {
    handle(record.getValue());
} catch (Exception e) {
    redisTemplate.delete(idemKey); // 失败回滚标记，允许重试
    throw e;
}
```

这里有个细节：幂等标记与业务处理严格说不是原子的，高并发下极端情况可能有空窗。真正强一致的场景（如资金），应该把幂等判断和业务写入放到同一个数据库事务里（比如用唯一索引防重），Redis 标记只当快速前置过滤。分层防护，才能既快又稳。

再补一个关于顺序性的实战要点。单个 Stream 内部消息是严格有序的（按 ID 递增），但一旦一个消费组里起了多个消费者并行消费，整体就无法保证处理顺序了——消息 A 分给 consumer-1、消息 B 分给 consumer-2，谁先处理完不一定。如果你的业务要求同一实体（如同一订单）的消息严格有序，常见做法是“按 key 分流”：把同一 orderId 的消息总是路由到同一个 Stream 或同一个消费者，以方向上的单线程处理换取局部有序。这和 Kafka 靠 partition key 保证分区内有序是一个道理。全局严格有序加高并发本身就是个矛盾，任何 MQ 都只能在分片粒度上保证有序。

监控上也有几个必盯的指标。一是 Stream 长度（`XLEN`），反映堆积；二是消费组的 lag，即“还没消费的消息数”，Redis 7.0 的 `XINFO GROUPS` 直接给出了 lag 字段，之前的版本要自己算；三是 PEL 长度（`XPENDING` 概况里的待确认数），它持续变大就说明消费端处理失败或漏 ACK，是个很好的故障早期信号。我们把这三个指标都接到告警里，lag 超阈告警“消费跟不上”，PEL 持续增长告警“有消息卡住”，效果很好。

在 Redis Cluster 下用 Stream 还有一个坑：一个 Stream 只会落在一个槽（slot）上，也就是单个节点上，无法自动分片。如果单个 Stream 流量极大，会把那个节点压成热点。解法是在业务层手动分片，比如按 orderId 取模拆成 `order:stream:0` 到 `order:stream:15` 多个 Stream（用 hash tag 控制分布），消费端并行监听多个。这已经有点在自己实现 Kafka 的 partition 了，也从侧面提醒你：当你开始需要手动分片时，往往就是该考虑专业 MQ 的信号了。

最后给 Stream 的常用命令做个速查小结：`XADD` 生产、`XLEN` 看长度、`XRANGE`/`XREVRANGE` 按范围回溯、`XREAD` 无组消费、`XGROUP` 管组、`XREADGROUP` 组内消费、`XACK` 确认、`XPENDING` 查待处理、`XCLAIM`/`XAUTOCLAIM` 认领转移、`XINFO` 看流和组的详细状态（排查问题时 `XINFO GROUPS` 特别有用）。把这套命令玩熟，你就能用 Redis 搭出一个具备可靠投递、故障转移、消息回溯能力的轻量级消息队列。

## 11.7 Redis队列与专业MQ的取舍边界

把 Stream 讲完，免不了要回答那个灵魂问题：既然 Stream 这么强，那还要 Kafka、RocketMQ 干什么？我结合几次真实的选型经历，把边界说清楚。先确立一个大前提：不要把“能不能用”和“适不适合用”混为一谈。很多东西 Redis 都能硬做出来，但能做不等于该做，工程上真正値钱的是“用最小的复杂度解决当前问题”这个判断力。

先看 Redis Stream 真正胜任的场景。一是你已经部署了 Redis，不想为了中等量的异步需求再引入一套重型中间件；二是消息量在日均千万级以内，不需要海量长期堆积；三是对延迟敏感，Redis 内存操作的低延迟比磁盘型 MQ 更有优势；四是运维团队小，养不起一套 Kafka 集群的人力。典型如中小型系统的订单异步处理、短信邮件发送、内部事件总线、实时排行榜更新等。

再看必须上专业 MQ 的场景。一是消息量巨大（日均亿级以上），Kafka 的磁盘顺序写加零拷贝在吞吐上碾压 Redis；二是需要 TB 级长期堆积和回溯（比如数据管道、日志收集），Redis 内存成本负担不起；三是需要复杂的消息治理能力，如事务消息、顺序消息、消息轨迹、死信队列、消息回放等，RocketMQ 这类开箱即用而 Redis 都得自己造；四是需要多语言、多团队、强治理的企业级生态。

还有一个很现实的因素是团队熟悉度。如果你们团队已经有人熟 Kafka、有现成的运维和监控体系，那哪怕量不大，直接用 Kafka 也无可厚非，因为它的边际成本对你们很低；反之，如果团队只有几个人、没人运维过 MQ，那 Redis Stream 就是更稳当的起点。选型从来不只是技术参数的对比，还得把团队能力这个变量算进去。

我的经验法则是：从 Redis Stream 起步，当你开始频繁遇到“得自己实现分片”“得自己写死信队列”“得自己做消息轨迹”这些需求时，就说明你已经在拿 Redis 仿写一个粗糙版的 MQ 了，这时迁移到专业 MQ 反而更省心。不要提前为了可能永远不会到来的量去上 Kafka，也不要因为情怀把已经明显超出 Redis 能力的业务硬扛在 Stream 上。工程上没有银弹，只有匹配当前阶段的正确选择。

## 总结

这一章我们把 Redis 做消息队列的三条路径彻底走了一遍。开篇先立起消息可靠性的三层标尺——不丢、不重、有序，这是评价一切消息队列的统一标准。然后逐一拆解：List 用 `LPUSH` + `BRPOP` 实现简单有序的点对点队列，但没有 ACK、丢消息、不能多消费组，只适合轻量异步任务，且务必用 `BRPOPLPUSH`/`BLMOVE` 加备份队列来兜底可靠性；Pub/Sub 提供广播能力，实时性好，但无持久化、发完即焚、消费者一离线就丢消息，只能用于配置推送、缓存失效这类“丢了不心疼”的通知场景，绝不能碰订单支付这类核心链路；Stream 才是 Redis 官方的正解，用只追加日志 + 消息组 + PEL + `XACK` 把持久化、回溯、多消费组、故障转移全部补齐，通过 `XCLAIM`/`XAUTOCLAIM` 实现消息不丢的故障转移。

几个核心结论请务必记住。第一，选型先问可靠性等级：能丢就 Pub/Sub，点对点轻量就 List，要可靠就 Stream，这是第一原则。第二，List 方案的命门是没有 ACK，`BRPOP` 弹出即删除，必须用备份队列 + 补偿任务才能防丢。第三，Pub/Sub 的本质是“至多一次”，任何依赖它做可靠投递的设计都是错的。第四，Stream 是“至少一次”，重复投递不可避免，消费端幂等是硬性要求，漏了迟早出资损事故。第五，Stream 用 `XADD` 别忘了 `MAXLEN ~` 控制内存，`XREADGROUP` 别忘了 `XACK`，这两个是最高频的翻车点。第六，Stream 强但不是万能，日均消息量、堆积需求、治理复杂度上到一定量级，该上 Kafka/RocketMQ 就别硬扛。

再分享一个容易被忽略的实战建议：不管你用哪种方案，上线前一定要专门演练三个异常场景：生产者正常发但消费者全部宕机（看消息会不会丢、堆积会不会撞内存）、消费者处理到一半被 kill（看消息能不能重投、会不会重复）、Redis 主从切换（看消费者会不会假死、能不能自愈）。这三个场景在生产环境都会真实发生，提前演练过的团队，真出事时才不会深夜手忙脚乱。消息队列的可靠性不是写出来的，是演练出来的。

给你一份可落地的选型决策清单，下次直接对照：一、可靠性要求高、要 ACK、要多消费组、要故障转移 → Redis Stream（首选）；二、一条消息广播给多方、且能容忍丢失 → Pub/Sub；三、单消费组、点对点、轻量异步任务、能接受补偿 → List + BRPOPLPUSH；四、海量消息、TB 级堆积、复杂路由与治理 → 直接上专业 MQ，别用 Redis 硬撑。

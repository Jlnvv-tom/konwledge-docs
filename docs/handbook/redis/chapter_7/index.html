Redis内存管理与淘汰策略

Redis之所以能够提供极高的读写性能，内存管理是其中最核心的底层支撑之一。作为一个纯内存数据库，Redis的每一条数据都完整地存储在内存中，内存的分配、释放、碎片整理以及容量上限控制，都直接影响着Redis的稳定性与吞吐能力。很多生产环境的Redis故障，追根溯源都与内存管理机制理解不深、配置不当有关。本章将从Redis内存模型开始，系统性地解析内存占用的各个组成部分，讲解maxmemory与淘汰策略的工作原理与配置方法，并结合实战经验提供监控与调优的实用技巧。

7.1 Redis内存模型：内存占用的核心组成部分​

理解Redis的内存占用，不能仅看used_memory一个数字。Redis的内存消耗由多个部分构成，弄清楚每一部分的来源与含义，才能在面对内存压力时做出准确的判断。

进程基本开销

Redis进程本身运行所需的基础内存开销包括代码段、运行时栈、全局变量等。这部分内存相对固定，通常在几十MB级别，不会随着数据量增长而明显变化。需要关注的是Redis运行时申请的各类内存分配子系统的开销，这部分体现在used_memory_peak和used_memory_peak_rss的差值上——前者是Redis实际申请的业务数据内存，后者是操作系统为进程分配的物理内存（包含未回收的碎片空间）。

数据内存

这是Redis内存占用的主体，即所有键值对数据本身占用的空间。具体包括：键名（key）本身的字符串、键值（value）对象的结构、对象头信息以及数据内容本身。以一个简单的字符串键SET mykey "hello"为例，Redis内部会为其分配一个sdshdr结构（存储字符串长度和可用空间）、一个redisObject对象头（存储类型、编码方式、LRU时钟等元数据）以及实际的数据内容。这三部分加起来才是该键的实际内存开销，远不是简单的字符串长度。

子进程内存

当Redis执行RDB快照持久化或AOF重写时，如果使用了操作系统的写时复制（Copy-On-Write）机制，父进程会fork出一个子进程。在fork的瞬间，子进程与父进程共享物理内存页，只有当某页内容被修改时才会真正复制出新页。因此在持久化操作期间，used_memory_rss会明显增长，增长幅度取决于数据被修改的比例。这是一个极易引发生产事故的坑点：如果在持久化期间有大量写操作，且maxmemory设置接近饱和，fork出的子进程可能因物理内存不足而被OOM Killer杀掉。一个典型的表现是Redis进程突然退出，日志中出现OOM-killing字样。

客户端缓冲区

每个Redis客户端连接都会分配输入缓冲区和输出缓冲区。输入缓冲区用于暂存客户端发送的尚未处理的命令，输出缓冲区用于暂存待返回给客户端的响应结果。正常情况下这两个缓冲区都很小，但一旦客户端处理速度跟不上服务端响应速度（比如客户端网络阻塞、客户端崩溃未及时关闭连接），输出缓冲区就会不断膨胀。如果大量客户端同时出现这种情况，缓冲区总占用可能达到数GB。Redis通过client-output-buffer-limit参数对单客户端输出缓冲区做了硬限制，但输入缓冲区没有限制，属于潜在隐患。

Lua脚本缓冲区

Redis在执行EVAL/EVALSHA命令时，会为Lua脚本分配一个脚本缓冲区。如果同一个脚本被多次加载但从未清除（通过SCRIPT FLUSH），Redis会保留多个脚本副本。正常业务中这个问题不明显，但如果使用动态生成脚本的架构且缺乏清理机制，脚本缓冲区会逐渐积累无用内容。

统计信息与辅助数据

Redis的INFO memory命令输出中还包括aof_rewrite_buffer_size（AOF重写缓冲区）、lua_caches（Lua脚本缓存）、startup_consumed_memory（启动时预分配的内存）等细项。这些项通常较小，但在精确分析内存构成时不应忽略。

通过执行INFO memory，可以一次性看到Redis内存消耗的全景数据：

7.2 内存限制配置：maxmemory参数与实际应用设置​

Redis提供了maxmemory参数来限制Redis进程可以使用的最大内存量。合理设置这个值是生产环境运维中最基础也是最重要的操作之一。

maxmemory的基本配置

在redis.conf中，配置maxmemory非常简单：

maxmemory 2gb

也可以通过命令行参数在启动时指定：

redis-server --maxmemory 2g

如果不设置maxmemory，Redis默认不限制内存使用，理论上会一直使用到操作系统可用内存耗尽为止。这个默认值在生产环境中极其危险——一旦Redis进程占满系统内存，操作系统会触发OOM Killer，强制杀掉Redis进程。因此，无论任何时候，都强烈建议显式设置maxmemory。

设置多大才合理？

maxmemory的值不应当等于服务器的全部物理内存。操作系统本身需要内存，Redis的fork子进程需要预留空间（通常为数据量的1到2倍，取决于写压力），其他可能运行在该机器上的进程也需要内存余量。经验法则如下：

对于纯Redis单实例部署，通常将maxmemory设置为机器物理内存的50%到70%。如果机器上同时运行Redis和其他服务，需要给其他服务留出至少2GB到4GB的余量，并将Redis的maxmemory设置在剩余空间的60%到80%范围内。

对于需要执行RDB快照的Redis实例，考虑到fork子进程在写密集场景下的内存膨胀，应当将maxmemory设置得更保守一些，通常在物理内存的40%到50%之间。在一次大规模数据写入的同时触发RDB快照，fork出的子进程可能短暂占用接近父进程数据量级别的额外内存。如果maxmemory设置过高，系统内存会迅速耗尽，OOM Killer可能在毫秒级内介入。

对于开启AOF持久化且使用appendfsync always的场景，每次写操作都会触发磁盘同步，进程可能在同步期间阻塞，此时如果有大量并发写入，内存会持续累积。AOF + always模式下的maxmemory应当比RDB模式设置得更低。

overcommit_memory的影响

除了maxmemory，Linux系统的内存分配策略overcommit_memory对Redis的fork行为有显著影响。该参数有三个值：0表示启发式内存超额分配、1表示始终允许超额分配、2表示永远不允许超额分配超过物理内存加swap总量。

当overcommit_memory设置为0时，Linux会尽量估算是否能够分配内存，如果估算失败会拒绝分配。如果Redis在fork时需要分配大块连续内存，而系统估算失败，fork可能返回ENOMEM错误，导致RDB快照失败或AOF重写失败。生产环境推荐将overcommit_memory设置为1，这是Redis官方文档推荐的值：

sysctl vm.overcommit_memory=1
echo "vm.overcommit_memory=1" >> /etc/sysctl.conf

同时确保maxmemory设置合理，两者的配合才能保证Redis在持久化期间不会因为内存问题而崩溃。

动态调整maxmemory

Redis 4.0之后支持通过CONFIG SET在运行时动态修改maxmemory：

127.0.0.1:6379> CONFIG SET maxmemory 3gb
OK

动态调整需要注意一点：如果将maxmemory调小到当前实际已用内存以下，Redis会立即开始淘汰数据，直到实际占用低于新的上限。如果不希望触发即时淘汰，应当先将maxmemory调大或保持不变，然后通过添加新节点、做数据迁移等方式逐步降低内存使用，再调整maxmemory。

设置过小的坑

曾经遇到过一个案例：开发团队将maxmemory设置为512MB，而实际业务数据加上Redis运行开销很快突破了这个限制。结果是Redis频繁触发淘汰，导致大量热数据被清除，读缓存命中率从99%骤降到60%，后端数据库压力暴涨。这个问题的根源在于maxmemory是根据部署时的数据快照估算的，但随着业务增长，键的数量和值的平均大小都增加了，而maxmemory却从未调整过。正确的做法是建立内存使用监控，当used_memory持续超过maxmemory的70%时发出告警，并及时评估是否需要扩容。

7.3 核心内存淘汰策略：volatile-lru、allkeys-lru等原理​

当Redis的内存使用达到maxmemory上限后，如果继续写入数据，Redis必须腾出空间来容纳新数据。这个腾出空间的过程就是内存淘汰（Eviction）。Redis提供了多达8种的淘汰策略，每种策略都有不同的适用场景和行为特点。

LRU系列策略

LRU（Least Recently Used，最近最少使用）是最常用的淘汰策略思想。Redis在每个redisObject结构中存储了一个24位的LRU时钟字段，当淘汰发生时，Redis会优先驱逐那些最久未被访问的键。

需要特别注意的是，Redis的LRU实现并非严格意义上的精确LRU算法，而是基于采样的近似LRU。Redis默认每次淘汰时会随机抽取maxmemory-samples（默认5）个键，从中选择最久未使用的那个。这个设计是为了在内存占用和CPU消耗之间取得平衡。精确LRU需要对所有键按照访问时间排序，这在键数量庞大的场景下CPU开销是不可接受的。maxmemory-samples越大，淘汰结果越接近精确LRU，但CPU开销也越高。Redis 4.0之后将默认值从5提升到了5（但文档中标注为16更准确，实际以源码为准），对于追求更精确淘汰效果的业务，可以适当调大：

maxmemory-samples 16

volatile-lru策略只会从设置了过期时间的键（TTL）中执行LRU淘汰，不会影响永久键。allkeys-lru策略则从所有键中执行LRU淘汰，不考虑是否设置了过期时间。

为什么会有这个区别？这是因为在实际业务中，有两种典型的缓存场景。一种是将Redis作为带过期时间的缓存使用，数据本身是有意设计为临时存储的，此时适合用volatile-lru：Redis会只从那些"可以过期"的键中淘汰，将永久键完整保留。另一种是将Redis作为唯一的数据存储（虽然Redis官方并不推荐这种用法），所有数据都希望被平等对待，此时适合用allkeys-lru：无论键是否设置了过期时间，都会参与淘汰。

LFU系列策略

LRU策略有一个固有的缺陷：某些键可能只是偶然被访问一次（比如系统启动时的初始化数据、一次性批处理脚本），但在LRU眼中它们被标记为"最近访问"，从而逃脱淘汰。结果是真正的高频热数据反而可能被淘汰。LFU（Least Frequently Used，最不经常使用）策略就是为了解决这个问题而设计的。

LFU基于访问频次来淘汰数据：访问次数越少的键越优先被淘汰。如果一个键每小时被访问1000次，另一个键每小时只被访问10次，后者在LFU策略下会优先被淘汰。Redis在LFU模式下使用两个字段来记录访问频次信息：lru字段的高16位存储上次递减计数器的时间戳（以分钟为单位），低8位存储访问计数。这种设计使得计数器可以根据时间自动衰减，防止一个键在历史某个时刻被高频访问后就一直占据"高频"位置。LFU的衰减逻辑由lfu-decay-time参数控制，表示计数器减半所需的时间（分钟）：

lfu-decay-time 1

设置为1表示如果一个键在1分钟内没有被访问，访问计数器就会减半。volatile-lfu和allkeys-lfu分别对应从有过期时间的键和所有键中执行LFU淘汰。

TTL和随机策略

volatile-ttl策略从设置了过期时间的键中选择TTL（Time To Live，剩余生存时间）最短的键优先淘汰。换句话说，越是快要过期的数据，越先被踢出。这种策略适合那些你希望优先保留"生命期较长"数据的场景，但实际使用频率不高。

volatile-random和allkeys-random分别是TTL和全量键随机淘汰。前者从有过期时间的键中随机选，后者从所有键中随机选。这两种策略通常只在调试或特殊场景下使用，随机淘汰的不确定性太高，难以保证缓存命中率。

noeviction策略

noeviction是默认策略，也是最安全的策略：Redis在达到内存上限后不再接受任何写入操作，读操作正常执行，但写入操作会返回错误：

(error) OOM command not allowed when used memory > maxmemory

这种策略适合将Redis纯粹作为不可变数据存储的场景，或者将maxmemory设置得非常大而实际数据量远未达到的场景。缺点是写入会直接失败，如果应用没有妥善处理这个错误，可能导致业务请求报错。

理解策略的行为差异

下面通过一个简化的示例来展示不同策略在面对同一数据集时的行为差异：

假设当前Redis中有三个键：key1（有过期时间，TTL=100秒）、key2（有过期时间，TTL=1000秒）、key3（无过期时间）。所有键都被访问了相同次数。现在需要淘汰1个键：

策略被淘汰的键
volatile-lrukey1（TTL最短）
allkeys-lrukey3（LRU最旧）
volatile-lfukey1（访问频次最低）
allkeys-lfukey3（访问频次最低）
volatile-ttlkey1（TTL最短）
volatile-random随机选择有过期时间的键
allkeys-random随机选择任意键

在实际生产中，allkeys-lru是最常用的策略，因为它模拟了一个"容量有限的缓存"的行为——当缓存满了，最近最少使用的数据被淘汰，腾出空间给新数据。

7.4 淘汰策略的选择依据：业务场景与数据特性匹配​

淘汰策略的选择不是拍脑袋决定的，而是应该基于业务数据的访问模式、数据的生命周期特征以及对缓存一致性的要求来综合判断。

场景一：读多写少的缓存型业务

这是Redis最典型的使用场景。大量读请求命中Redis缓存，极少量写请求更新缓存和数据库。在这种场景下，数据可以被分为"热点数据"和"冷数据"。热点数据访问频率极高，冷数据被访问的频率很低甚至只访问一次。

对于这种场景，推荐使用allkeys-lru。无论数据是否设置了过期时间，Redis都会持续淘汰最近最少使用的键，保证热点数据始终留在内存中。如果使用volatile-lru，那些没有设置过期时间但实际上已经"冷掉"的键会永久占用内存，直到手动删除或Redis重启。这是一种常见的配置错误：开发者给部分键设置了过期时间（希望它们自动清理），给部分键没有设置过期时间（认为它们应该永久保留），然后配置了volatile-lru，结果是没有设置过期时间的永久键不断积累，最终耗尽内存。

场景二：需要区分临时数据和永久数据的业务

有些业务需要在Redis中同时存储两类数据：一类是真正的永久存储（比如分布式锁的锁信息、会话数据），另一类是临时缓存（比如排行榜的实时排名、限流的计数窗口）。此时推荐volatile-lru配合合理的过期时间设计：永久数据不使用EXPIRE命令，因此不参与淘汰；临时数据统一设置合适的TTL（比如1小时、24小时），参与LRU淘汰。

但这个方案有一个隐患：如果程序逻辑错误导致永久数据没有正确地不设置过期时间（比如代码中每次SET都会覆盖EXPIRE），永久数据也会被TTL标记，从而参与淘汰，导致永久数据丢失。因此更好的做法是在架构层面将永久数据和临时数据分别存储在不同的Redis实例中，各自使用不同的淘汰策略。

场景三：热点集中、存在流量突刺的业务

某些业务存在明显的流量突刺，比如秒杀活动、热点事件推送。这类场景的特点是短时间内大量新数据涌入，流量峰值过后这些数据的访问量骤降。对于这种场景，volatile-lru配合较短的TTL（如5到30分钟）是合适的：大量短期数据设置了过期时间，在TTL到期后自动被淘汰，不需要依赖LRU算法来决策；allkeys-lru在流量突刺期间可能因为大量新键涌入而导致热数据被意外淘汰，缓存命中率下降。

场景四：对缓存命中率要求极高的业务

如果缓存命中率对业务有决定性影响（比如搜索推荐、广告排序等），建议评估allkeys-lfu策略。LFU策略会记录每个键的访问频次，淘汰策略倾向于保留高频键，即使高频键刚刚被写入。只要一个键被多次访问，它就会在淘汰竞争中占据优势。对于缓存预热后的稳态运行，LFU通常比LRU能维持更高的命中率。

但LFU也有缺点：如果一个键在预热阶段没有被充分访问（比如刚上线的新商品），它在LFU眼中是"低频"键，即使它即将成为热键，也可能在初期被淘汰。解决这个问题需要在业务层面做预热，或者使用LRU+LFU混合策略（目前Redis尚未原生支持，需要业务层自行实现）。

场景五：严格不容许数据丢失的写入场景

如果Redis中的每一条数据都是不可丢失的重要数据，比如分布式锁信息、分布式ID生成器的状态，应当使用noeviction策略并设置足够大的maxmemory，确保内存永远不会成为写入瓶颈。同时配合主从复制架构，将数据实时同步到从节点，以防主节点发生故障导致数据丢失。这种场景下使用任何一种淘汰策略都是不恰当的，因为任何淘汰都意味着数据永久消失。

策略选择决策树

综合以上分析，可以根据以下决策树来选择淘汰策略：

第一，确认Redis的角色定位。如果是缓存（数据可丢失），进入下一步；如果是持久存储（数据不可丢失），选择noeviction。

第二，确认是否需要区分永久键和临时键。如果所有键都是平等的，使用allkeys-lru；如果需要区分，使用volatile-lru或按不同实例分离存储。

第三，确认是否需要基于访问频次。频繁出现"一次性访问键占用内存"场景时，选择allkeys-lfu或volatile-lfu。

第四，评估TTL策略是否适用。如果数据天然具有明确的生命周期（如会话30分钟、排行榜1小时），使用volatile-ttl。

7.5 内存碎片的产生与解决：碎片率监控与内存整理​

内存碎片是Redis运维中一个非常隐蔽但影响重大的问题。很多时候Redis报告的used_memory和操作系统观察到的used_memory_rss（Resident Set Size）之间存在明显差距，这个差距就是内存碎片。碎片率高会导致明明数据量不大，Redis却占用了大量物理内存，严重时会导致服务器内存耗尽。

内存碎片是如何产生的

Redis默认使用jemalloc作为内存分配器。jemalloc将内存划分为多个不同大小的块（chunk），当应用程序请求分配内存时，jemalloc会分配一个恰好大于等于请求大小的块。例如，应用程序请求分配100字节，jemalloc可能实际分配128字节的块。看似浪费了28字节，但这是为了减少内存碎片和提高分配效率。

问题在于：Redis的数据是动态增删改的。当一个字符串从10字节扩展到200字节时，如果当前块空间不足，Redis会重新分配一个更大的块来存储新的内容，原有的块被释放。此时，释放的块可能比200字节小得多，无法容纳后续200字节级别的数据请求，从而成为碎片空闲块。随着数据的不断更新和删除，这些无法利用的小碎片会逐渐积累。

碎片率的计算与监控

Redis通过mem_fragmentation_ratio（碎片率）来量化内存碎片程度：

mem_fragmentation_ratio = used_memory_rss / used_memory

当碎片率大于1.0时，说明存在内存碎片。碎片率在1.0到1.5之间通常认为是正常范围，由jemalloc的分配策略决定。如果碎片率超过1.5，说明有超过50%的已分配物理内存是未被有效利用的碎片，此时内存效率极低。如果碎片率超过2.0，运维人员必须立即介入处理。

INFO memory的输出中包含以下相关指标：

127.0.0.1:6379> INFO memory | grep -E "fragment|allocator"
allocator_frag_ratio:1.23
allocator_frag_bytes:2621440
allocator_rss_ratio:1.50
allocator_rss_bytes:5242880
mem_fragmentation_ratio:1.73
mem_fragmentation_bytes:7340032

其中allocator_frag_ratio表示jemalloc内部的碎片率，allocator_rss_ratio表示物理内存分配与Redis实际使用之间的比例。mem_fragmentation_ratio是两者的综合结果。

主动内存整理：ACTIVEDEFRAG

Redis 4.0引入了主动内存碎片整理功能（Active Defragmentation），通过activedefrag yes参数启用。该功能在后台定期检查内存碎片的分布情况，并在发现有足够多的碎片空间可以回收时，自动对数据键进行重新分配和整理。

activedefrag yes
# 以下为可选调优参数
activedefrag-ignore-bytes 100mb      # 碎片总大小超过100MB时开始整理
activedefrag-threshold-lower 10     # 碎片率超过10%时开始整理
activedefrag-threshold-upper 100    # 碎片率超过100%时强制整理
activedefrag-cycle-min 25           # 整理占用的最小CPU时间比例（%）
activedefrag-cycle-max 75           # 整理占用的最大CPU时间比例（%）

activedefrag的工作原理是在后台线程中对数据键进行小范围的增量重分配。每次处理一个数据键时，检查其当前所在内存块是否是最优大小，如果是则跳过，如果否则分配一个更紧凑的块并迁移数据。这个过程是增量进行的，不会一次性消耗大量CPU资源，从而避免影响Redis的正常服务。

启用activedefrag后，Redis会自动在后台整理碎片，无需人工干预。但需要注意，主动碎片整理本身会消耗一定的CPU和内存（迁移过程中需要临时分配新块）。在已经内存紧张的服务器上，开启碎片整理可能导致情况进一步恶化，因此建议在内存使用率低于70%时启用此功能。

被动碎片处理：重启

在没有开启主动碎片整理的情况下，重启Redis是最直接也是最彻底解决碎片问题的方法。Redis重启后重新加载数据，jemalloc会重新分配内存，碎片率会重置为接近1.0。但生产环境中不能随意重启Redis，尤其是一个承载着大量读写请求的主节点。正确的做法是：

首先确认碎片率的严重程度。如果碎片率在1.5到2.0之间，可以通过从业务低峰期进行主从切换（故障转移），让从节点升级为主节点，然后在原主节点上重启后重新加入集群作为新的从节点完成数据同步。如果碎片率超过2.0且used_memory_rss接近物理内存上限，应当立即告警并启动紧急扩容流程，同时在业务低峰期进行节点重启。

碎片监控的最佳实践

建议在监控系统中将mem_fragmentation_ratio纳入告警规则：当碎片率超过1.5时发送警告，超过2.0时发送严重告警。同时记录allocator_frag_ratio和allocator_rss_ratio，以便区分是jemalloc内部碎片还是物理内存分配层面的碎片，后者可能与操作系统层面的内存管理问题有关。

# 在监控脚本中使用redis-cli获取碎片率
frag_ratio=$(redis-cli -h 127.0.0.1 -p 6379 INFO memory | \
  grep "mem_fragmentation_ratio" | awk -F: '{print $2}' | tr -d '\r')

if (( $(echo "$frag_ratio > 1.5" | bc -l) )); then
  echo "WARNING: Redis memory fragmentation ratio is $frag_ratio"
fi

7.6 实战优化：Redis内存占用的监控与调优技巧​

理论知识最终要落地到实践。本节从监控体系建立、常见内存问题排查和具体优化手段三个方面，分享生产环境中经过验证的内存优化经验。

建立完整的内存监控体系

监控是优化的前提。Redis自身提供的INFO命令已经包含了大量有价值的内存指标，关键是要建立持续性的监控和趋势分析。以下是需要重点关注的内存指标及其告警阈值建议：

used_memory与maxmemory的比率是最核心的指标。当used_memory / maxmemory超过70%时，建议发出预警；超过85%时，发出严重告警并考虑扩容或清理数据；超过95%时，Redis随时可能开始大量淘汰数据或拒绝写入。这个比率可以通过redis-cli INFO memory | grep used_memory获取，并配合定时任务写入监控系统。

mem_fragmentation_ratio碎片率如前所述，超过1.5需要关注，超过2.0必须处理。

evicted_keys（淘汰键数量）记录了自Redis启动以来因内存限制而被淘汰的键总数。如果这个值在持续增长，说明maxmemory设置偏小，或者业务数据的增长超出了当初的预期。结合expired_keys（过期键数量）和keyspace_hits / keyspace_misses（缓存命中/未命中），可以综合判断缓存是否工作正常。

client_biggest_input_buf和client_longest_output_list反映了单个客户端消耗的最大缓冲区大小。如果这个数字异常大，说明存在处理速度严重不匹配的客户端，需要检查网络链路或客户端代码。

一个实用的监控脚本整合示例：

#!/bin/bash
# Redis内存健康检查脚本
REDIS_HOST="127.0.0.1"
REDIS_PORT="6379"
INFO=$(redis-cli -h $REDIS_HOST -p $REDIS_PORT INFO memory)

used_mem=$(echo "$INFO" | grep "^used_memory:" | awk -F: '{print $2}')
max_mem=$(redis-cli -h $REDIS_HOST -p $REDIS_PORT CONFIG GET maxmemory | tail -1)
frag_ratio=$(echo "$INFO" | grep "mem_fragmentation_ratio:" | awk -F: '{print $2}')
evicted=$(echo "$INFO" | grep "evicted_keys:" | awk -F: '{print $2}')

usage_ratio=$(echo "scale=4; $used_mem / $max_mem" | bc)
echo "当前内存使用率: $(echo "$usage_ratio * 100" | bc)%"
echo "内存碎片率: $frag_ratio"
echo "累计淘汰键数量: $evicted"

if (( $(echo "$usage_ratio > 0.85" | bc -l) )); then
  echo "WARNING: 内存使用率超过85%，请尽快处理"
fi

Big Key问题：内存的无声杀手

Big Key是指单个键的值占用内存过大的键。比如一个STRING类型的键存储了几百MB的数据，或者一个ZSET/SET/HASH类型的键包含了几百万个成员。Big Key会引发多个层面的问题。

在内存层面，一个巨大的STRING键会占用连续的数百MB内存块，如果需要淘汰这个键，被释放的空间可能无法被有效利用（碎片化）。同时，一次性读取或删除Big Key会产生巨大的网络传输和CPU开销。以一个包含100万个成员的ZSET为例，如果直接执行ZRANGE key 0 -1，Redis需要一次性返回所有成员的数据，网络带宽瞬间被打满。对于这种ZSET，ZRANGE key 0 -1 BYSCORE LIMIT 0 100分批获取，每次处理100个成员，可以显著降低单次操作的资源消耗。

识别Big Key的方法是使用redis-cli --bigkeys命令：

redis-cli -h 127.0.0.1 -p 6379 --bigkeys

该命令会扫描Redis中的所有键，按照STRING/LIST/SET/ZSET/HASH类型分别找出内存占用最大的键，并输出统计摘要。在离线分析阶段运行这个命令，可以快速定位需要拆分或优化的Big Key。

解决Big Key的原则是拆分和压缩。对于ZSET类型，可以按时间或ID范围进行分桶存储，比如将一个包含所有用户分数的ZSET拆分为多个小ZSET，每个ZSET存储一段时间或一个区间的数据。对于STRING类型的大值，如果数据有规律，可以考虑用压缩算法（如zstd）压缩后再存储，或者改用Hash类型将大值拆分为多个field。对于HASH类型，可以使用 HSCAN 命令渐进式遍历，而不是一次性获取全部字段。

键数量膨胀的治理

键数量膨胀是指Redis中存储了大量小型键，导致键名本身的内存开销占比过高。每个键都有一个redisObject对象头（至少16字节），加上键名字符串本身，以及过期时间等元数据。即使一个键只存储一个字节的值，其元数据开销也可能达到几十字节。

键数量膨胀通常由以下原因导致：程序设计时将数据库中的行列直接映射为Redis的键值对，导致每个字段一行变成了Redis中的一个键。例如，在关系型数据库中，一个用户表有100个字段，在Redis中错误地设计为每个用户100个键（user:{id}:field1、user:{id}:field2等），而不是使用一个HASH键（user:{id}包含100个field）。前者的键数量是后者的100倍，元数据开销也是100倍。

治理键数量膨胀的核心思路是重新审视数据模型设计。优先使用Hash、List、ZSET等聚合类型代替大量独立String键。对于已经存在的膨胀键，可以通过SCAN + DELETE + 重新写入的方式分批迁移，注意迁移期间需要暂停写入或使用双写策略保证一致性。

过期键的及时清理

设置了过期时间的键并不一定会立即从内存中消失。Redis使用惰性删除（Lazy Expired）和定期删除（Periodic Delete）两种机制来清理过期键。

惰性删除是指当客户端尝试访问一个键时，Redis检查该键是否已过期，如果已过期则立即删除并返回空值。这种方式节省CPU资源，但如果过期键一直不被访问，就永远不会从内存中清除。如果大量键都设置了相同的过期时间且过期后无人访问，内存会在过期后仍然被占用很长时间。

定期删除是指Redis每100毫秒随机抽取一批设置了过期时间的键，检查并删除其中已过期的键。随机抽取避免了对全部键的遍历，保持了CPU开销的可控性。但随机策略意味着有些过期的键可能连续多次未被抽中，一直留在内存中。

对于需要精确控制的场景，可以在业务层面主动触发过期数据的清理。比如一个限流场景使用了ZSET存储时间窗口数据，可以用定时任务定期清理窗口之外的历史数据：

# 清理7天前的限流记录（假设使用ZSET，score为时间戳）
redis-cli -h 127.0.0.1 -p 6379 \
  --eval "redis.call('ZREMRANGEBYSCORE', KEYS[1], '-inf', ARGV[1])" \
  ratelimit:user:123 \
  $(date -d '7 days ago' +%s)

连接数与内存的间接关系

不要忽视Redis连接数对内存的影响。每个TCP连接都会消耗文件描述符和少量内存，连接数过万时这个开销累积起来也相当可观。如果应用存在连接泄漏（创建了连接但未正确关闭），连接数会持续增长，每个连接都附带输入输出缓冲区，内存压力随之增加。通过INFO clients查看当前连接的客户端数量，如果connected_clients持续增长，说明存在连接泄漏。

127.0.0.1:6379> INFO clients
# Clients
connected_clients:1523
client_longest_output_list:0
client_biggest_input_buf:2097152
blocked_clients:0

client_biggest_input_buf:2097152表示当前最大输入缓冲区为2MB。如果这个数字持续增大，说明有客户端发送了大量待处理命令，需要检查客户端代码或网络链路是否存在异常。

确认maxmemory已设置且不超过物理内存的70%，并根据持久化方式和机器负载留有充分余量。maxmemory不为零时，maxmemory-policy必须设置了合理的淘汰策略，否则Redis将在内存满时拒绝写入。

键数量和键平均大小在合理范围内，不存在大量短生命周期的键未设置TTL或TTL设置过长导致内存浪费。

客户端连接数稳定，不存在连接泄漏，输出缓冲区大小正常。

总结​

本章系统梳理了Redis内存管理的完整知识体系。从内存模型的角度，Redis的内存消耗由数据内存、进程开销、子进程内存、客户端缓冲区和脚本缓冲区等多个部分构成，理解每个部分的特点是精确诊断内存问题的前提。maxmemory参数是控制Redis内存上限的总开关，设置时必须充分考虑持久化开销、系统余量和业务增长。8种淘汰策略覆盖了从LRU到LFU、从TTL到随机等各种淘汰需求，选择策略的核心依据是数据是否需要区分永久和临时、是否需要基于访问频次做出淘汰决策。内存碎片是容易被忽视但影响严重的问题，mem_fragmentation_ratio超过1.5就应当考虑启用主动碎片整理或进行节点重启。实战调优的核心在于建立完善的监控体系，及时发现Big Key、键数量膨胀、连接泄漏等问题，并结合数据模型优化从根本上降低内存消耗。

## 8.2 避坑指南：8个常见部署与使用陷阱

**坑1：盲目追求大模型**

| 误区 | 正解 |
|------|------|
| 一定要跑70B模型 | 7B-14B模型在大多数任务上够用 |
| 量化会严重降低质量 | INT4量化在对话场景几乎无感知 |
| 参数越多越聪明 | 模型能力≠参数量，数据质量更重要 |

**坑2：忽视显存瓶颈**

| 误区 | 正解 |
|------|------|
| 8GB显存想跑FLUX.1 | 用FP8量化版或换SDXL |
| 同时跑多个模型 | 串行跑，用完释放 |
| 不用Tiled VAE | 大图必须开Tiled VAE |

**坑3：Docker端口冲突**

| 误区 | 正解 |
|------|------|
| 多个工具都用3000端口 | 部署前规划端口表 |
| 直接docker run | 用docker-compose统一管理 |

**坑4：模型下载从Hugging Face**

| 误区 | 正解 |
|------|------|
| 直接从HF下载（国内） | 用ModelScope或HF镜像 |
| 下载完整模型 | 按需下载指定文件 |

**坑5：知识库文档不预处理**

| 误区 | 正解 |
|------|------|
| 直接扔原始PDF | 先清洗→分段→去噪 |
| 不调chunk参数 | 根据文档类型调chunk size和overlap |
| 只用一种检索方式 | 混合检索（关键词+向量） |

**坑6：n8n工作流过度复杂**

| 误区 | 正解 |
|------|------|
| 一个工作流做所有事 | 按场景拆分，模块化设计 |
| 不加错误处理 | 每个关键节点加try-catch |
| 不测试直接上线 | 先用测试数据跑通 |

**坑7：安全裸奔**

| 误区 | 正解 |
|------|------|
| 不改默认密码 | 首次部署第一步改密码 |
| 开放到公网 | 内网部署+VPN访问 |
| 不备份数据 | 定期备份Docker卷 |

**坑8：工具囤积症**

| 误区 | 正解 |
|------|------|
| 装了30个工具只用3个 | 先装核心3-5个，深入用3个月 |
| 每个工具都浅尝 | 一个工具用透，再扩展 |
| 追逐每个新工具 | 关注趋势但不频繁切换 |

> 工具的价值在于使用深度，而非数量。怕浪猫用过100+AI工具，真正每天在用的不超过8个。

## 8.3 趋势观察：端侧AI、Agent化与MCP协议

**趋势1：端侧AI爆发**

| 趋势 | 说明 | 影响 |
|------|------|------|
| Mac统一内存 | M系列芯片可跑大模型 | Mac成为AI工作站 |
| 手机端AI | 骁龙8 Gen4端侧7B | AI不依赖云端 |
| 浏览器AI | Chrome内置Gemini Nano | 网页级AI能力 |
| 开源小模型 | 1B-3B模型质量提升 | 低端设备也能跑 |

**趋势2：Agent化**

| 趋势 | 说明 | 影响 |
|------|------|------|
| 从对话到执行 | AI不只是聊天，而是完成任务 | Cline/OpenClaw类工具爆发 |
| 多Agent协作 | 多个AI代理协同工作 | 复杂任务自动化 |
| 工具调用标准化 | Function Calling/MCP协议 | 工具间互操作性提升 |
| 自主决策 | AI自己规划步骤 | 减少人工干预 |

**趋势3：MCP协议**

MCP（Model Context Protocol）是Anthropic提出的开放协议，标准化AI模型与外部工具的连接方式。

| 方面 | 说明 |
|------|------|
| 是什么 | AI模型与工具之间的"USB接口" |
| 解决什么 | 每个工具不用单独写适配层 |
| 当前状态 | 2025年已有数百个MCP服务器 |
| 影响 | 工具生态互联互通，降低集成成本 |

**趋势4：开源追赶闭源**

| 时间节点 | 预测 |
|---------|------|
| 2025年底 | 开源模型在多数任务上接近GPT-4水平 |
| 2026年 | 开源Agent框架成熟，自主完成任务 |
| 2027年 | 开源多模态模型普及，端侧可用 |
| 长期 | 开源成为AI基础设施，闭源转向高端场景 |
